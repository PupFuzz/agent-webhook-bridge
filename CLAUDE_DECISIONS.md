# Decision log

> **Append-only.** Every load-bearing decision in the bridge gets an entry here, with the rationale + alternatives considered + consequences, so a future session can reconstruct the why of each design choice. Corrections to a prior decision get a **new** entry titled "Correction to DL-NNN"; the original stays frozen.
>
> This repository's history begins at **v0.12.0**. An earlier implementation existed (a Python consumer + standalone PHP receiver) but is not part of this repo; only the decisions that are load-bearing for the current code are recorded here.

## DL-001 — Single synchronous Laravel app: receive → classify → stage → dispatch, in one request

- **Date:** 2026-05-29
- **Context:** The bridge gives an AI agent (or operator) real-time visibility into kanban-board activity — and any other webhook-emitting upstream following the same shape — without burning tokens polling. It runs as one install per agent, co-located with its database on a single host. The core design question is whether to decouple the fast "receive" path from the slower "process" path with a durable queue, or to do the whole thing inline.
- **Decision:** One Laravel 13 app, **synchronous in-request dispatch**. A webhook is HMAC-verified in middleware (`VerifyHmacSignature`, constant-time over the raw body) and size-bounded (`EnvelopeSizeLimit`); the per-provider `WebhookAdapter` parses the envelope; then `DispatchService` runs **classify → stage to `inbox.jsonl` → run handlers** synchronously in the same request, returning `200` only when every subscribed agent has been processed. **No queue worker, no consumer cron, no daemon.** `webhook_events` is a dedup gate (`UNIQUE(delivery_id)`) plus an audit/replay store — not a work queue; `agent_dispatches` is the per-agent, per-event outcome ledger.
- **Three-way failure treatment** (the load-bearing reliability contract, in `WebhookController` + `DispatchService`):
  - **(A) classify throws** → record the error on that agent's dispatch (`processed_at` stays null), ack **200**. A classifier bug must not wedge delivery; the raw event is stored and `bridge:replay <id>` re-runs it.
  - **(B) inbox staging throws** → propagate → **5xx** → kanban-board redelivers. `inbox.jsonl` is the durable pull-backstop; silently losing a staged intent is the one unacceptable outcome.
  - **(C) handler throws** → mark that agent's dispatch *done-with-note* and continue. Per-agent isolation: one agent's channel server being down must not fail the delivery or the other agents.
- **At-least-once is borrowed, not built.** Any internal/durability failure throws → Laravel returns `5xx` → kanban-board's webhook retry redelivers; `inbox.jsonl` is the durable pull-backstop that `bridge:inbox` reads even if a push never arrived. The dispatch write primitive is `dedupCreate` (create + catch `UniqueConstraintViolationException` → refetch), so a redelivery resumes safely. There is deliberately **no** `DB::transaction` around the dispatch loop — a handler does network I/O (channel_push) and a rollback cannot un-send a POST; each dispatch records its own outcome independently.
- **Alternatives considered:**
  - **A durable queue + worker / scheduled command.** Rejected: the receiver and the processor run on the *same host* against the *same database*, so a queue would decouple two things that are never actually decoupled in time or space — while adding a daemon/cron failure surface (silent stalls, lock contention, lag). If a future install genuinely needs to absorb a receive/process throughput mismatch, Laravel's queue is a localized change behind the same `DispatchService` — re-introduce it then, with evidence, not speculatively.
  - **Wrapping the dispatch loop in a DB transaction.** Rejected: a handler making an external call inside a transaction holds a connection across network I/O, and a rollback can't un-send a push. There is no all-or-nothing semantic to protect.
  - **Pure polling** (cron against the changelog API). Rejected for token cost + latency; the webhook push with an `inbox.jsonl` pull-backstop is strictly better.
- **Consequences:** One PHP runtime (Laravel 13 / PHP 8.3 / Eloquent over MariaDB 10.6+, SQLite for tests). Per-`(provider, scope)` HMAC secrets live under `BRIDGE_SECRET_DIR`; the DB password in `.env`; per-agent policy in `~/.config/agent-webhook-bridge[-prod|-dev]/<agent>.yml`. Latency is the handler runtime (single-digit ms for the shipped `log_intent` / `channel_push`), comfortably under kanban-board's delivery timeout — a genuinely slow handler is the signal to reach for a queue, not a reason to pre-build one. CI is `Laravel Tests` (PHPUnit on a SQLite + MariaDB 10.6/11 matrix) + `Security` (gitleaks); static analysis is `phpstan-laravel.neon` (level 7, scoped to `app/Bridge`).

## DL-002 — Agent recognition keys on immutable numeric ids; GitHub `sender.id`, not `sender.login`; shared accounts declared once

- **Date:** 2026-05-30
- **Context:** `AgentRegistry` translated a webhook's raw actor id into a friendly agent name across two axes — kanban `user_id` (an immutable DB integer) and GitHub `sender.login` (the **mutable** username). An incident exposed the fragility: the GitHub account behind a multi-agent deployment was renamed. The PAT and the immutable numeric account id were unchanged and everything authenticated fine, but every event now carried the new login, which matched nothing in `agents.json` — recognition went stale and echo-suppression-by-login silently broke, requiring a hand-edit in N places (the same login was denormalized across every agent entry). Root cause: recognition keyed on a renameable identifier, denormalized N times. The numeric account id is immutable and is present on every event (`sender.id`).
- **Decision:** Recognition keys on **immutable numeric ids only**, and matching is **provider-aware**.
  - `GitHubAdapter` extracts `actor_id` from `sender.id` (numeric), not `sender.login`. `AgentRegistry::actorFromEvent(string $provider, …)` consults the kanban axis for kanban events and the github axis for github events — so a kanban `user_id` and a github `sender.id` that are the same integer never cross-match (a hazard the old "try one untyped string against both lookups" code would have had once both axes became numeric).
  - `agents.json` `schema_version` → **2**. Per-agent identity is `kanban_user_id` + `github_user_id` (both immutable ints). A GitHub account **shared** by multiple agents is declared **once** under a top-level `shared_identities[]` block (`{github_user_id, github_login?, agents[]}`) — the explicit, declared-once form of the shared-login collision bypass: a shared account resolves to `Actor.name = null` on purpose so a custom classifier (FROM:-line / repo-scope) re-attributes. `null` name + `isKnownAgent = false` is preserved byte-for-byte so that re-attribution layer is untouched.
  - `github_login` becomes a **display-only label** (never a matching key). When an event's `sender.login` differs from the configured label for that id, the registry logs a one-line drift warning (`configured github_login "X" is stale; account <id> is now "Y"`), deduped once per login per request. Recognition is unaffected — a rename is now a **non-event**.
  - **No backward-compatibility path** (single-operator project): a `schema_version: 1` file is not parsed — `load()` warns with a migration message and degrades to an empty registry (classifiers still run; echo-by-raw-id still works). The login matching axis (`byGithubLogin`) is removed, not deprecated.
- **Alternatives considered:**
  - **Keep `github_login` as a matching key, add `github_user_id` alongside (ID wins).** Rejected: keeps a renameable identifier in the hot path and a dual-format `if (id) … else login …` branch the Single-Format-Principle warns against. The operator explicitly waived backward-compat, so there's no reason to carry the mutable axis forward.
  - **Resolve the current login from the id at runtime via the GitHub API (cached) for display.** Rejected (scope): adds an API-token dependency, a cache, and failure modes for a purely cosmetic gain. For the shared-account case the display name is classifier-assigned anyway. A static label + free drift warning gives the operator the actionable signal without the machinery.
  - **Per-agent `github_user_id` repeated on each entry, relying on collision-detect to bypass.** Rejected as the *primary* shared-account mechanism: it re-creates the N-place denormalization the incident was about. Collision-detect is retained as a safety net for *accidental* duplicate ids (warns, points at `shared_identities`); the intentional shared account is declared once.
- **Consequences:** `webhook_events.actor_id` stores the immutable numeric id for GitHub events (better for replay/audit). `treat_as_echo_ids` for a GitHub identity now uses the numeric `github_user_id`, which survives renames. Adding a provider that authenticates by a mutable handle should follow the same rule: carry the immutable id in `actor_id`, keep the handle as a display label. Files: `app/Bridge/Support/AgentRegistry.php` + `RegisteredAgent.php` + `SharedIdentity.php`, `app/Bridge/Adapters/GitHubAdapter.php`, `app/Bridge/Dispatch/DispatchService.php` (passes `$provider` into `actorFromEvent`).

## DL-003 — Same-event ReactionTarget coalescing is enforced; cross-delivery debounce is not

- **Date:** 2026-05-30
- **Context:** `ReactionTarget` carries `debounceKey` (defaults to `targetId`) and `debounceSeconds`, and `ClassifyResult` / `ReactionTarget` docblocks + `docs/reaction-target-schema.json` + the handler-log all described `debounce_key` coalescing. But the v0.12 rewrite dropped the implementation: `DispatchService` iterated `$result->targets` and invoked **every** target's handler — no coalescing anywhere. A divergent-duplication audit surfaced the doc-vs-code gap. The original async design had two debounce behaviors: same-pass coalescing (collapse duplicate keys) **and** a cross-pass `DebounceTracker` time-window (the `debounceSeconds` window) that suppressed repeats across drain passes.
- **Decision:** Split the two behaviors by what the synchronous model needs.
  - **Same-event coalescing — restored** (`DispatchService`): within one `ClassifyResult`, targets sharing a `debounceKey` collapse last-wins (associative-array key) before the handler loop, so a classifier emitting N targets to one bucket fires that handler once. Cheap, matches the long-published contract (the field, its `targetId` default, the docblocks, the schema, the handler-log key), and protects the bridge's core extension point — a custom classifier emitting duplicate-bucket targets no longer double-fires a handler.
  - **Cross-delivery debounce — intentionally NOT enforced.** The `debounceSeconds` time-window / old `DebounceTracker` was an async-drain construct (suppress repeats across batched passes). The synchronous model handles one event per request with no batching; redelivery dedup is the `webhook_events` `UNIQUE(delivery_id)` gate + kanban-board's retry (DL-001), not a time window. `debounceSeconds` remains **advisory metadata** carried to the handler + handler-log for a consumer that chooses to honor it.
- **Alternatives considered:**
  - **Document-only (demote `debounce_key` to advisory, no coalescing).** Rejected: leaves a published coalescing key that silently doesn't coalesce — a footgun precisely at the custom-classifier extension point, and the cheaper-looking option only because it ships less correctness.
  - **Re-implement the cross-delivery `DebounceTracker` (time-window suppression).** Rejected: reintroduces persistent cross-request state the synchronous design (DL-001) deliberately removed; the UNIQUE-gate + upstream retry already cover redelivery.
- **Consequences:** `ClassifyResult.targets` are coalesced last-wins by `debounceKey` at dispatch time; built-in classifiers emit ≤1 target so are unaffected, but custom classifiers can rely on `debounceKey` to dedupe their own same-event targets. Files: `app/Bridge/Dispatch/DispatchService.php` (the coalescing loop), `ClassifyResult.php` + `ReactionTarget.php` (docblocks), `docs/reaction-target-schema.json`.

## DL-004 — HandlerRegistry is a container singleton so custom handlers register through the documented extension point

- **Date:** 2026-05-30
- **Context:** `docs/customization.md` advertised custom-handler registration as `HandlerRegistry::register(name, instance)` from a `ServiceProvider`, implying the registry is resolvable from the container. But `BridgeServiceProvider` built `DispatchService` with a `new HandlerRegistry` constructed *inside* the bind closure — never bound to the container. So `app(HandlerRegistry::class)` / `afterResolving(HandlerRegistry::class, …)` could never reach the instance the dispatch loop actually used; the only working path was re-binding `DispatchService` wholesale and duplicating its constructor wiring (subscriptions / agents / registries), which is fragile across upgrades. A peer integrator reported the only documented handler-extension point as non-functional. (Filed alongside a separate, low-priority echo FR — per-agent self-echo for a shared upstream identity — tracked independently.)
- **Decision:** Bind `HandlerRegistry` as a container **singleton** in `BridgeServiceProvider`; `DispatchService` resolves it from the container (`$this->app->make(HandlerRegistry::class)`) instead of constructing its own. Operators register custom handlers via `afterResolving(HandlerRegistry::class, fn ($r) => $r->register('x', new XHandler))` in a `ServiceProvider` — the documented path now works against the **exact** instance the dispatcher uses, with no provider-ordering requirement (the callback is registered at boot, before the first webhook resolves the registry). `HandlerRegistry` carries no per-request state (the four shipped handlers are stateless), so a per-process singleton is correct — unlike the config-derived `SubscriptionRegistry` / `AgentRegistry`, which is exactly why `DispatchService` itself stays `bind` (rebuilt per request from `config('bridge.config_dir')`).
- **Alternatives considered:**
  - **Config-driven handler map** (`config/bridge.php => 'handlers' => ['x' => XHandler::class]` merged at build time). Rejected as the fix: it is a *second* registration mechanism beyond the reported bug, and a `name => FQCN` map cannot express handlers that need constructor dependencies (an API client, a configured path) — which the `afterResolving` path expresses naturally as `new XHandler($dep)`. The per-agent `classifier.class` extension is config-driven because it is per-agent; handlers are one global registry, so DI is the better fit. Can be added later if declarative, dependency-free registration becomes desirable.
  - **Re-bind `DispatchService` wholesale** (the prior documented workaround). Rejected: it duplicates the bridge's constructor wiring in operator code, so any future change to `DispatchService`'s dependencies silently breaks every operator override.
- **Consequences:** `HandlerRegistry` is a process singleton (built once per FPM worker, mirroring the `ClassifierResolver` per-process instance cache); `afterResolving` fires once on first resolution and the dispatcher reuses that instance. Tests that construct `DispatchService` directly with `new HandlerRegistry` are unaffected (they bypass the container). Files: `app/Providers/BridgeServiceProvider.php`, `docs/customization.md` (registration walkthrough + the extension-point table), `tests/Feature/Providers/BridgeServiceProviderTest.php` (regression test).

## DL-005 — Per-agent echo for a shared upstream identity via `ClassifyResult::reattributedActor`

- **Date:** 2026-05-30
- **Context:** `shared_identities` (DL-002) lets several agents run under one upstream account (e.g. four agents sharing one GitHub `sender.id` because the platform forces shared credentials). For such an account the registry resolves `Actor.name = null` on purpose, deferring attribution to a custom classifier. But echo suppression runs **before** `classify()` and matches on `(actor.name | actor.id)`, so for a shared identity the only working lever is `treat_as_echo_ids` on the raw id — which is **all-or-nothing**: list the id and *every* agent's view of the account is suppressed (the whole inbox dies); omit it and no self-echo happens at all (each agent sees its own writes echoed back). There is no per-agent middle ground, because the true author isn't known until the classifier re-attributes — and the classifier (resolved per-FQCN, shared across agents) is never told which agent it's serving. A peer integrator running the cross-cutting "subscribes-to-everything" agent reported every comment/PR it posts pinging back into its own inbox: continuous wasted attention + a latent self-react risk, defeating a guarantee the bridge advertises. DL-002 had already named this re-attribution layer as the intended completion point ("`null` name preserved byte-for-byte so that re-attribution layer is untouched").
- **Decision:** Add an optional `?Actor $reattributedActor` to `ClassifyResult` (additive, defaults null). A classifier that recovers the true author (FROM:-line / repo-scope / any operator convention) returns it there; **after** `classify`, `DispatchService` re-runs the **same** per-agent echo check (`EchoSuppression::default` over `identity.self` + `treat_as_echo` + `treat_as_echo_ids`) on the reattributed actor and marks the dispatch done (suppressed) when it's the serving agent's own write. The classifier reports *who authored the event* (content analysis); the dispatcher decides *is that me?* (per-agent policy) — so one shared classifier instance yields per-agent self-echo, a different shared-id agent's write still surfaces (its recovered name isn't the serving agent's self), and classifiers still **do not filter** (the documented invariant holds — they re-attribute, the dispatcher filters). Null `reattributedActor` is a no-op: every shipped classifier leaves it null, so behaviour is unchanged unless an operator opts in.
- **Alternatives considered:**
  - **Pass `AgentConfig` / `selfIdentity` into `classify()` (the FR's option 1).** Rejected: it is a **breaking** change to the `Classifier` contract (every custom classifier + both shipped ones must change signature, and it breaks the deliberate v0.11 Python-parity of the signature), and it moves the echo *decision* into each classifier — contradicting the documented "echo suppression has already happened — do not re-filter inside classify" invariant and inviting per-agent branching inside a per-FQCN-cached instance. It is strictly more powerful (a classifier could shape intents per agent), but that power isn't needed for this gap and costs the contract + the separation of concerns. The additive field expresses exactly the need with zero breakage.
  - **A second, separate echo-config concept evaluated post-classify.** Rejected: there is no new policy to express — the agent's existing `identity.self` / `treat_as_echo` already say "what is me." Reusing the same predicate on a better-attributed actor is the whole point; a parallel config would be a second source of truth for one question.
  - **Mutate `Intent.actor` and have the dispatcher re-derive the author from the intents.** Rejected: echo is an *event-level* decision (one event, one true author) and must also cover the target-only / no-intent result; reading it off per-intent actors is ambiguous and misses those cases. A single top-level field on the result is unambiguous.
  - **Defer (document as a known gap).** Rejected: it leaves a feature the bridge ships (`shared_identities`) holed for every shared-identity deployment, the fix is small and proportionate (one optional field + one dispatcher branch), and it's evidence-driven (a concrete deployment feels the pain now) rather than speculative.
- **Consequences:** `ClassifyResult` gains a third optional constructor arg; `DispatchService` gains one post-classify branch (reusing `isEcho`). The pre-classify echo gate is unchanged — the post-classify check is a refinement that fires only when attribution improved. The **`treat_as_signal` allowlist is intentionally NOT re-run** post-classify here (the FR scoped to echo; a shared-identity + non-empty `treat_as_signal` interaction — where the null-name event is dropped pre-classify before it can be re-attributed — is a separate, currently-unexercised concern, left for when a deployment hits it). A bridge-shipped re-attribution helper (a base classifier doing FROM:-line parsing) is a possible future DX follow-up, deliberately out of scope. Files: `app/Bridge/Dispatch/ClassifyResult.php`, `app/Bridge/Dispatch/DispatchService.php`, `docs/customization.md` (§ Per-agent echo for a shared upstream identity), `docs/multi-agent.md` (Path C), `tests/Feature/Dispatch/DispatchServiceTest.php` + `tests/Fixtures/ReattributingClassifier.php`.

## DL-006 — Per-agent inbox surfacing for a single multi-agent install

- **Date:** 2026-05-30
- **Context:** A single install fans out to N agents (the "Acme" topology: a cross-cutting `pm` + scope-bounded agents), but surfacing was shared-only: `bridge:inbox` read one `inbox.jsonl` for all agents, staged lines carried no serving-agent field, and there was no `--agent` selector. One `FROM:device` coordination event fanned out to three agents → `pm`'s inbox surfaced it ~3× with no way to filter. The v0.11 bridge had per-agent surfacing via `bin/inbox --config <agent>`; the v0.12 rewrite dropped it. The topology that needs it can't use the one-install-per-agent answer (standing rule 7): one public web server, two agents are separate OS users on the same box with no web server, one agent is a remote firewalled host. Reported by a peer integrator.
- **Decision:** Make per-agent surfacing first-class, all additive/opt-in (default behaviour unchanged).
  - **Agent-attributed intents.** Every staged inbox line carries the serving `agent` (distinct from `actor`, the event's author). `BRIDGE_INBOX_LAYOUT=shared|per-agent|both` (default `shared`) selects where lines land: shared `inbox.jsonl`, per-agent `inbox-<agent>.jsonl`, or both.
  - **`bridge:inbox --agent <name>`** surfaces only that agent's unseen intents (its per-agent file, or the shared file filtered by the `agent` tag), with a per-agent seen cursor `inbox-seen-<agent>.json` so one agent marking-seen never hides another's. `BRIDGE_DEFAULT_AGENT` lets a single-primary install keep a bare `bridge:inbox`. `--agent` also added to `bridge:stats` / `bridge:inspect`.
  - **Cross-user read** via a documented group convention: `BRIDGE_STATE_DIR` (new; defaults to `config_dir/state`) points runtime state OUTSIDE the 0700 secret-holding `config_dir`, and `BRIDGE_INBOX_GROUP` + `BRIDGE_INBOX_FILE_MODE` (default `0640`) set the group + mode on per-agent inbox files so a co-located different-OS-user agent reads its own `inbox-<agent>.jsonl` directly (no artisan-as-that-user). **`BRIDGE_INBOX_GROUP` requires `per-agent` layout** (`bridge:check` + dispatch fail-closed otherwise) — under shared/both a group-traversable state dir would expose the shared `inbox.jsonl` (every agent's intents) to the group. The bridge sets perms on the per-agent *files* only and never widens the directory (the operator's `install -d -m 0750 -g` does that), so it can't silently expose a dir holding other agents' state; perms failures are logged, not swallowed or 5xx'd. Seen-cursors stay install-user-owned (only the bridge:inbox process writes them).
  - **`channel.route_intents`** (per-agent YAML, with `channel.url` for the SSH-tunneled remote case): the dispatcher synthesizes a `channel_push` per staged intent and merges it into the existing coalescing + best-effort (treatment-C) handler loop — the config-driven form of `EventDrivenClassifier`, so a remote/idle agent's intents reach it without hand-coded targets. Namespaced debounceKey (`channel_push:<subject>`) so it never clobbers an unrelated classifier target.
  - **Cursor-advance reliability fix.** `bridge:inbox` now advances the seen cursor ONLY when the output can reach a consumer (a hook event that supports `additionalContext`, or a manual non-hook run). On `Stop`/`Notification`-type hooks (plain stdout the model never sees) it prints but leaves intents unseen, so the next `SessionStart`/`PreToolUse` surfaces them — wiring `bridge:inbox` on `Stop` no longer silently eats intents. `--no-cursor-advance` forces a non-advancing peek.
- **Alternatives considered:**
  - **Custom surface formatter (the v0.11 mechanism).** Rejected: surfacing layout is core, not an agent-specific extension; per-agent files + an `--agent` selector are simpler and make multi-reader / cross-OS-user consumption trivial (a reader tails its file).
  - **State under the existing `config_dir/state` for cross-user.** Rejected: `config_dir` is `0700` because it holds HMAC secrets + tokens — it can't be group-traversable without exposing them. A separate `BRIDGE_STATE_DIR` is required for safe cross-user read.
  - **Classifier-emitted `channel_push` only (no routing config).** Kept as the option for per-push headers (e.g. a Bearer token), but `route_intents` covers the common "just deliver my intents to my channel" case without code.
- **Consequences:** Inbox lines gain an additive `agent` field (pin-to-schema consumers fail-soft). New env: `BRIDGE_STATE_DIR`, `BRIDGE_INBOX_LAYOUT`, `BRIDGE_DEFAULT_AGENT`, `BRIDGE_INBOX_FILE_MODE`, `BRIDGE_INBOX_GROUP`. New per-agent YAML: `channel.url`, `channel.route_intents`. Pair `route_intents` with a plain inbox classifier, not `EventDrivenClassifier`, or each event pushes twice. Files: `app/Bridge/Dispatch/IntentLog.php` + `DispatchService.php`, `app/Bridge/Support/BridgePaths.php` + `AgentConfig.php`, `app/Bridge/Handlers/ChannelPushHandler.php`, `app/Console/Commands/Bridge/{Inbox,Stats,Inspect}Command.php`, `config/bridge.php`, `.env.example`, docs `multi-agent.md` / `multi-host.md` / `consumer-guide.md`, tests across those. (A follow-up config-schema-v2 will simplify the per-agent YAML further — drop `identity.self`, hoist per-install settings, fold identity into the YAMLs.)

## DL-007 — Config schema v2: filename-as-name, per-install settings hoisted, identity folded into the YAMLs

- **Date:** 2026-05-30
- **Context:** The per-agent config surface had accreted redundancy that a senior-dev review surfaced. The agent name was denormalized across FOUR sites with nothing enforcing agreement (the `<agent>.yml` filename, `identity.self`, the `agents.json` `name`, and `treat_as_echo: [self]`) — a latent echo-break bug if `identity.self` ever drifted from the filename. Per-INSTALL settings (`receiver.base_url`, `api.<provider>.base_url`) were identical for every agent yet repeated in every per-agent YAML (4× in the Acme topology). `agents.json` duplicated the agent roster the YAMLs already define. The agent's own id was denormalized between `agents.json` and a hand-listed `treat_as_echo_ids`. `channel.name` was dead (no runtime reader). No backcompat constraint applies (pre-1.0, single operator).
- **Decision:** A breaking config-schema v2.
  - **The filename IS the canonical agent name** (and its echo "self"). `identity.self` is removed; `DispatchService` keys echo on `agentName`.
  - **Per-install settings hoisted to `config/bridge.php`/`.env`:** `BRIDGE_RECEIVER_BASE_URL`, and a `providers` map for API base URLs (`BRIDGE_KANBAN_API_BASE_URL`; github defaults to `api.github.com`). Only the per-agent token stays per-agent.
  - **Per-agent identity folded into the YAML** (`identity: {kanban_user_id, github_user_id, github_login}`); `AgentRegistry::fromAgentConfigs` builds the roster by scanning the YAMLs the `SubscriptionRegistry` already reads. `agents.json` is replaced by an optional `shared-identities.json` holding ONLY `shared_identities` (absent when no account is shared); the `schema_version`/v1-migration machinery is deleted.
  - **`BRIDGE_DIR`** collapses `BRIDGE_CONFIG_DIR`+`BRIDGE_SECRET_DIR` (both overridable). **API token by convention** `<secret_dir>/<provider>/token` (`TokenPath::for`, sibling to `SecretPath::for`), with a per-agent `api.<provider>.token_path` override — keyed on (secret_dir, **provider**), NOT agent, because a token is a per-account credential shared by shared-identity agents (the review caught an agent-keyed convention as wrong). **`channel.name` deleted** (dead field).
  - **Self echo-ids auto-seeded** from the agent's own `identity` ids — `echo_suppression: {}` is the common case; operators never hand-list self ids (this applies DL-002's own lesson to the bridge's config and removes the self-id drift footgun). `treat_as_echo`/`treat_as_signal` name OTHER agents only.
  - **Two fail-closed decisions (operator-approved):** (a) identity + subscription reads are unified on the one scanned-YAML source, so a malformed YAML 5xx's both (no more soft-degrade-to-empty `agents.json`); (b) a `treat_as_signal` name with no matching agent config throws at load (was warn-only) — a typo would otherwise silently classify everything NOT-IN-SIGNAL.
- **Alternatives considered:**
  - **Fold `shared_identities` into `config/bridge.php`** (the reviewer's suggestion, to avoid a lonely JSON sidecar). Rejected: `config/bridge.php` is repo-tracked, but shared-identity is per-install policy — putting it in a tracked file forces operators to edit tracked files and creates merge conflicts on `git pull`. A `shared-identities.json` in the (non-repo) config dir, alongside the YAMLs, is the right home for per-install policy.
  - **Convention-derive the token path keyed on agent.** Rejected (the review's catch): tokens are per-provider-account; shared-identity agents share one token, and the canonical install points multiple agents at one token file. Keyed on (secret_dir, provider), with a per-agent override.
  - **Keep `identity.self` / `api.base_url` / `receiver` for migration tolerance.** Rejected: no backcompat constraint; tolerating them would preserve the denormalization. They become unknown-key warnings (receiver) or ignored (api.base_url) on a migrated file, which is acceptable.
- **Consequences:** `<agent>.yml` shrinks to `identity` (ids) + `subscriptions` + optional `echo_suppression`/`classifier`/`channel`/`surface`/`api.token_path`. New `UrlValidator` (shared http-URL validation, used by `bridge:check` on the install endpoints) + `TokenPath`. `bridge:check` now validates: classifier FQCN resolvability, the inbox-surfacing config, install endpoint URLs, token-path readability, secret presence, an unknown `BRIDGE_DEFAULT_AGENT`, and reports `shared-identities.json`. A `BridgeCommand` base consolidates the `--option` coercion across the 6 `bridge:*` commands. `ProviderApiConfig` deleted. Operators migrate: move ids into each YAML's `identity`, drop `identity.self`/`receiver`/`api.base_url`/`channel.name`, set `BRIDGE_DIR`/`BRIDGE_RECEIVER_BASE_URL`/`BRIDGE_KANBAN_API_BASE_URL`, move the token to `<secret_dir>/<provider>/token`, and (if sharing an account) rename `agents.json` → `shared-identities.json` keeping only the `shared_identities` block. Files: `config/bridge.php`, `app/Bridge/Support/{AgentConfig,AgentRegistry,BridgePaths,TokenPath,UrlValidator}.php`, `app/Providers/BridgeServiceProvider.php`, `app/Bridge/Dispatch/DispatchService.php`, `app/Bridge/Support/SignalAllowlist.php`, all `app/Console/Commands/Bridge/*`, `.env.example`, `examples/sample-config/*`, docs.

## DL-008 — Config-level channel auth: `channel.auth.token_path` carries a Bearer token on the `route_intents` push

- **Date:** 2026-05-31
- **Context:** Phase-2 of the per-agent-surfacing work (DL-006). In the Acme topology a single multi-agent install fans out to a cross-cutting `pm` plus scope-bounded agents. `pm` (same OS user as the install) live-wakes over a `0600` UDS — `channel.socket` + `route_intents: true`, no token, because the socket's owner-only perms are the trust boundary. The two co-located impl agents (`backend`, `inventory`) are each a **different OS user** on the same box, so the install-user bridge can't reach a UDS they own — it must POST over loopback TCP (`channel.url`). A loopback TCP listener is **not** user-isolated like a `0600` UDS: any local account can POST to `127.0.0.1:<port>` and inject forged channel events into another agent's Claude session, making it act. So the channel server's `Authorization: Bearer <token>` is the **real trust boundary** there, not defense-in-depth. But the no-code `route_intents` path had no way to attach one: `DispatchService` synthesizes the routed `channel_push` as `payload: $intent->toArray()` (no headers); `ChannelPushHandler` falls back to the agent's `channel.socket`/`channel.url` for the *endpoint* but `buildHeaders()` sourced auth only from the payload; and `AgentConfig` had no channel-auth key. `Classifier::classify(...)` isn't passed the serving agent, so a shared classifier can't emit `backend`'s url+token vs `inventory`'s — the only classifier-side workaround was one bespoke subclass per agent hardcoding a port+token, duplicated logic differing by two constants. The remote firewalled `device` (SSH-tunnel topology) needs the identical capability. Reported by a peer integrator (phase-2 follow-on FR).
- **Decision:** Let an agent's YAML declare channel auth and have the `route_intents` path carry it. All additive/opt-in; absent key ⇒ today's behaviour.
  - **`channel.auth.token_path`** — a file path (chmod 600), not an inline secret, mirroring the `<secret_dir>/<provider>/token` + `api.<provider>.token_path` convention (standing rule 3). Nested under `auth` (not a flat `channel.token_path`) so a future non-Bearer scheme or custom header name is an **additive** key, not a breaking rename — the channel token is injected into an arbitrary third-party channel server's auth scheme, exactly where shape variance shows up, unlike the bridge-defined `api` token. The `auth.` segment also disambiguates it from the `url`/`socket` endpoint *locators* it sits beside.
  - **Read fail-closed at point-of-use, never in a payload.** `ChannelToken::read()` is the single reader: the file must exist, be readable, **not group/world-readable** (`mode & 0o077 == 0`, SSH-style), and be non-empty — else it throws. The handler attaches `Authorization: Bearer <token>` directly on the request and **never** places the token in `payload['headers']`. The routed payload is staged to `inbox.jsonl` and logged by other handlers; reading at point-of-use keeps the secret out of every serializable/logged structure by construction (verified: not in the synthesized payload, `inbox.jsonl`, `webhook_events`, `agent_dispatches`, nor the dispatch note — Laravel's `RequestException` summarizes the response, not request headers). The thrown message carries the token **path** only, never the value (the note is persisted).
  - **Perms enforcement is load-bearing, not hygiene.** On a multi-user host the Bearer token is the trust boundary precisely because any local account can POST to the port; a token file any local account can *read* lets them forge the header, collapsing the boundary. So the bridge refuses a non-0600 token at push time, not merely by convention. `bridge:check` warns at preflight (the file is operator-config, may not be provisioned yet — matching the existing API-token warn, not a hard fail), and the handler is fail-closed at dispatch.
  - **Applied only when the endpoint is agent-config-sourced** (the fallback branch — payload omitted both `socket` and `url`, which is exactly the `route_intents` synthesis). A classifier that hand-builds a target with its **own** `url` does NOT get the agent's token — it carries its own `headers` map. Predicate is "endpoint came from this agent's config", so a credential minted for the agent's endpoint never rides a target the classifier pointed elsewhere.
  - **`ChannelConfig` value object** replaces `resolveChannel()`'s positional return tuple (was `{?socket, ?url, route_intents}`, would have grown to 4 with two adjacent nullable strings) — matches the `EchoSuppressionConfig` DTO idiom (CLAUDE_CONVENTIONS) so adding the next channel field can't silently mis-assign at the destructuring site. `AgentConfig` holds one `channel` prop instead of four flat `channel*` props.
  - **`TokenFile::readTrimmed()`** is the shared trim/non-empty primitive; `ProvisionCommand::readToken` and `ChannelToken` both use it so the empty/whitespace edge can't drift (the API token stays lenient null-return; the channel token layers perms + throw on top).
- **Alternatives considered:**
  - **Synthesize the token into the routed `payload['headers']` in `DispatchService` (FR option 1).** Rejected: the routed payload is `$intent->toArray()`, which is staged/logged — putting a Bearer secret in a payload array is one refactor away from a log line or `inbox.jsonl`. Read-at-point-of-use keeps it out of serializable structures by construction.
  - **A general `channel.headers` map (arbitrary static headers).** Rejected for now: reintroduces an inline secret in YAML (against rule 3) for the only scheme that exists today (Bearer). The `auth.` namespace reserves room to add it later for a non-Bearer scheme without a breaking change — building it now is speculative (scope discipline).
  - **Apply the token to any classifier-emitted target missing `headers` (FR option 2's broader form).** Rejected: a classifier pointing at a different endpoint/port would get this agent's token injected — sending the credential somewhere it wasn't minted for. Scope to the config-endpoint fallback only.
  - **Symlink-reject the token path like `channel.socket` does.** Declined (documented): the socket path is *payload*-sourced (classifier-controllable) with a TOCTOU connect window; `token_path` is *operator-config*-sourced inside the `0700` config dir, so the 0600 owner-only perm is the defense — a file an attacker can't write can't be symlink-swapped by them.
  - **Flat `channel.token_path` for symmetry with `api.<provider>.token_path`.** Rejected: that symmetry copies the wrong property (flat shape) while ignoring the relevant one — the `api` token is bridge-defined and fixed-shape; the channel token feeds a third-party server's auth scheme. Nesting is also closer to `api.<provider>.token_path`'s actual depth.
- **Consequences:** New per-agent YAML key `channel.auth.token_path` (additive; closes the `multi-host.md` "use the classifier form when you need a Bearer header" limitation so the no-classifier path is usable on cross-user/multi-tenant hosts and the SSH-tunnel `device` case alike). The HTTP transport's token is enforced 0600; operators chmod the token file or the routed push errors (recorded, inbox backstop holds the intent). New invariant now security-load-bearing: **no global HTTP request logging** (it would capture the `Authorization` header) — there is none today. `channel.auth.token_path` is **rejected at config load unless `channel.url` is set** (the HTTP transport): a UDS channel's trust boundary is its `0600` filesystem perms, so socket+token is a contradiction caught fail-closed rather than silently ignored — keeping the token surface on the TCP transport only. On the fallback branch, config auth is authoritative: any payload-supplied `Authorization` header (case-insensitively) is dropped before ours is set, so a classifier-emitted fallback target can't smuggle a second Bearer. Files: `app/Bridge/Support/{ChannelConfig,ChannelToken,TokenFile,AgentConfig}.php`, `app/Bridge/Handlers/ChannelPushHandler.php`, `app/Bridge/Dispatch/DispatchService.php`, `app/Bridge/Exceptions/ChannelTokenException.php`, `app/Console/Commands/Bridge/{Check,Provision}Command.php`, `examples/sample-config/agent.yml.example`, `docs/multi-host.md`, `docs/multi-agent.md`, tests across handler/config/dispatch/support.
