## DL-001 â€” Single synchronous Laravel app: receive â†’ classify â†’ stage â†’ dispatch, in one request

- **Date:** 2026-05-29
- **Context:** The bridge gives an AI agent (or operator) real-time visibility into kanban-board activity â€” and any other webhook-emitting upstream following the same shape â€” without burning tokens polling. It runs as one install per agent, co-located with its database on a single host. The core design question is whether to decouple the fast "receive" path from the slower "process" path with a durable queue, or to do the whole thing inline.
- **Decision:** One Laravel 13 app, **synchronous in-request dispatch**. A webhook is HMAC-verified in middleware (`VerifyHmacSignature`, constant-time over the raw body) and size-bounded (`EnvelopeSizeLimit`); the per-provider `WebhookAdapter` parses the envelope; then `DispatchService` runs **classify â†’ stage to `inbox.jsonl` â†’ run handlers** synchronously in the same request, returning `200` only when every subscribed agent has been processed. **No queue worker, no consumer cron, no daemon.** `webhook_events` is a dedup gate (`UNIQUE(delivery_id)`) plus an audit/replay store â€” not a work queue; `agent_dispatches` is the per-agent, per-event outcome ledger.
- **Three-way failure treatment** (the load-bearing reliability contract, in `WebhookController` + `DispatchService`):
  - **(A) classify throws** â†’ record the error on that agent's dispatch (`processed_at` stays null), ack **200**. A classifier bug must not wedge delivery; the raw event is stored and `bridge:replay <id>` re-runs it.
  - **(B) inbox staging throws** â†’ propagate â†’ **5xx** â†’ kanban-board redelivers. `inbox.jsonl` is the durable pull-backstop; silently losing a staged intent is the one unacceptable outcome.
  - **(C) handler throws** â†’ mark that agent's dispatch *done-with-note* and continue. Per-agent isolation: one agent's channel server being down must not fail the delivery or the other agents.
- **At-least-once is borrowed, not built.** Any internal/durability failure throws â†’ Laravel returns `5xx` â†’ kanban-board's webhook retry redelivers; `inbox.jsonl` is the durable pull-backstop that `bridge:inbox` reads even if a push never arrived. The dispatch write primitive is `dedupCreate` (create + catch `UniqueConstraintViolationException` â†’ refetch), so a redelivery resumes safely. There is deliberately **no** `DB::transaction` around the dispatch loop â€” a handler does network I/O (channel_push) and a rollback cannot un-send a POST; each dispatch records its own outcome independently.
- **Alternatives considered:**
  - **A durable queue + worker / scheduled command.** Rejected: the receiver and the processor run on the *same host* against the *same database*, so a queue would decouple two things that are never actually decoupled in time or space â€” while adding a daemon/cron failure surface (silent stalls, lock contention, lag). If a future install genuinely needs to absorb a receive/process throughput mismatch, Laravel's queue is a localized change behind the same `DispatchService` â€” re-introduce it then, with evidence, not speculatively.
  - **Wrapping the dispatch loop in a DB transaction.** Rejected: a handler making an external call inside a transaction holds a connection across network I/O, and a rollback can't un-send a push. There is no all-or-nothing semantic to protect.
  - **Pure polling** (cron against the changelog API). Rejected for token cost + latency; the webhook push with an `inbox.jsonl` pull-backstop is strictly better.
- **Consequences:** One PHP runtime (Laravel 13 / PHP 8.3 / Eloquent over MariaDB 10.6+, SQLite for tests). Per-`(provider, scope)` HMAC secrets live under `BRIDGE_SECRET_DIR`; the DB password in `.env`; per-agent policy in `~/.config/agent-webhook-bridge[-prod|-dev]/<agent>.yml`. Latency is the handler runtime (single-digit ms for the shipped `log_intent` / `channel_push`), comfortably under kanban-board's delivery timeout â€” a genuinely slow handler is the signal to reach for a queue, not a reason to pre-build one. CI is `Laravel Tests` (PHPUnit on a SQLite + MariaDB 10.6/11 matrix) + `Security` (gitleaks); static analysis is `phpstan-laravel.neon` (level 7, scoped to `app/Bridge`).

## DL-002 â€” Agent recognition keys on immutable numeric ids; GitHub `sender.id`, not `sender.login`; shared accounts declared once

- **Date:** 2026-05-30
- **Context:** `AgentRegistry` translated a webhook's raw actor id into a friendly agent name across two axes â€” kanban `user_id` (an immutable DB integer) and GitHub `sender.login` (the **mutable** username). An incident exposed the fragility: the GitHub account behind a multi-agent deployment was renamed. The PAT and the immutable numeric account id were unchanged and everything authenticated fine, but every event now carried the new login, which matched nothing in `agents.json` â€” recognition went stale and echo-suppression-by-login silently broke, requiring a hand-edit in N places (the same login was denormalized across every agent entry). Root cause: recognition keyed on a renameable identifier, denormalized N times. The numeric account id is immutable and is present on every event (`sender.id`).
- **Decision:** Recognition keys on **immutable numeric ids only**, and matching is **provider-aware**.
  - `GitHubAdapter` extracts `actor_id` from `sender.id` (numeric), not `sender.login`. `AgentRegistry::actorFromEvent(string $provider, â€¦)` consults the kanban axis for kanban events and the github axis for github events â€” so a kanban `user_id` and a github `sender.id` that are the same integer never cross-match (a hazard the old "try one untyped string against both lookups" code would have had once both axes became numeric).
  - `agents.json` `schema_version` â†’ **2**. Per-agent identity is `kanban_user_id` + `github_user_id` (both immutable ints). A GitHub account **shared** by multiple agents is declared **once** under a top-level `shared_identities[]` block (`{github_user_id, github_login?, agents[]}`) â€” the explicit, declared-once form of the shared-login collision bypass: a shared account resolves to `Actor.name = null` on purpose so a custom classifier (FROM:-line / repo-scope) re-attributes. `null` name + `isKnownAgent = false` is preserved byte-for-byte so that re-attribution layer is untouched.
  - `github_login` becomes a **display-only label** (never a matching key). When an event's `sender.login` differs from the configured label for that id, the registry logs a one-line drift warning (`configured github_login "X" is stale; account <id> is now "Y"`), deduped once per login per request. Recognition is unaffected â€” a rename is now a **non-event**.
  - **No backward-compatibility path** (single-operator project): a `schema_version: 1` file is not parsed â€” `load()` warns with a migration message and degrades to an empty registry (classifiers still run; echo-by-raw-id still works). The login matching axis (`byGithubLogin`) is removed, not deprecated.
- **Alternatives considered:**
  - **Keep `github_login` as a matching key, add `github_user_id` alongside (ID wins).** Rejected: keeps a renameable identifier in the hot path and a dual-format `if (id) â€¦ else login â€¦` branch the Single-Format-Principle warns against. The operator explicitly waived backward-compat, so there's no reason to carry the mutable axis forward.
  - **Resolve the current login from the id at runtime via the GitHub API (cached) for display.** Rejected (scope): adds an API-token dependency, a cache, and failure modes for a purely cosmetic gain. For the shared-account case the display name is classifier-assigned anyway. A static label + free drift warning gives the operator the actionable signal without the machinery.
  - **Per-agent `github_user_id` repeated on each entry, relying on collision-detect to bypass.** Rejected as the *primary* shared-account mechanism: it re-creates the N-place denormalization the incident was about. Collision-detect is retained as a safety net for *accidental* duplicate ids (warns, points at `shared_identities`); the intentional shared account is declared once.
- **Consequences:** `webhook_events.actor_id` stores the immutable numeric id for GitHub events (better for replay/audit). `treat_as_echo_ids` for a GitHub identity now uses the numeric `github_user_id`, which survives renames. Adding a provider that authenticates by a mutable handle should follow the same rule: carry the immutable id in `actor_id`, keep the handle as a display label. Files: `app/Bridge/Support/AgentRegistry.php` + `RegisteredAgent.php` + `SharedIdentity.php`, `app/Bridge/Adapters/GitHubAdapter.php`, `app/Bridge/Dispatch/DispatchService.php` (passes `$provider` into `actorFromEvent`).

## DL-003 â€” Same-event ReactionTarget coalescing is enforced; cross-delivery debounce is not

- **Date:** 2026-05-30
- **Context:** `ReactionTarget` carries `debounceKey` (defaults to `targetId`) and `debounceSeconds`, and `ClassifyResult` / `ReactionTarget` docblocks + `docs/reaction-target-schema.json` + the handler-log all described `debounce_key` coalescing. But the v0.12 rewrite dropped the implementation: `DispatchService` iterated `$result->targets` and invoked **every** target's handler â€” no coalescing anywhere. A divergent-duplication audit surfaced the doc-vs-code gap. The original async design had two debounce behaviors: same-pass coalescing (collapse duplicate keys) **and** a cross-pass `DebounceTracker` time-window (the `debounceSeconds` window) that suppressed repeats across drain passes.
- **Decision:** Split the two behaviors by what the synchronous model needs.
  - **Same-event coalescing â€” restored** (`DispatchService`): within one `ClassifyResult`, targets sharing a `debounceKey` collapse last-wins (associative-array key) before the handler loop, so a classifier emitting N targets to one bucket fires that handler once. Cheap, matches the long-published contract (the field, its `targetId` default, the docblocks, the schema, the handler-log key), and protects the bridge's core extension point â€” a custom classifier emitting duplicate-bucket targets no longer double-fires a handler.
  - **Cross-delivery debounce â€” intentionally NOT enforced.** The `debounceSeconds` time-window / old `DebounceTracker` was an async-drain construct (suppress repeats across batched passes). The synchronous model handles one event per request with no batching; redelivery dedup is the `webhook_events` `UNIQUE(delivery_id)` gate + kanban-board's retry (DL-001), not a time window. `debounceSeconds` remains **advisory metadata** carried to the handler + handler-log for a consumer that chooses to honor it.
- **Alternatives considered:**
  - **Document-only (demote `debounce_key` to advisory, no coalescing).** Rejected: leaves a published coalescing key that silently doesn't coalesce â€” a footgun precisely at the custom-classifier extension point, and the cheaper-looking option only because it ships less correctness.
  - **Re-implement the cross-delivery `DebounceTracker` (time-window suppression).** Rejected: reintroduces persistent cross-request state the synchronous design (DL-001) deliberately removed; the UNIQUE-gate + upstream retry already cover redelivery.
- **Consequences:** `ClassifyResult.targets` are coalesced last-wins by `debounceKey` at dispatch time; built-in classifiers emit â‰¤1 target so are unaffected, but custom classifiers can rely on `debounceKey` to dedupe their own same-event targets. Files: `app/Bridge/Dispatch/DispatchService.php` (the coalescing loop), `ClassifyResult.php` + `ReactionTarget.php` (docblocks), `docs/reaction-target-schema.json`.

## DL-004 â€” HandlerRegistry is a container singleton so custom handlers register through the documented extension point

- **Date:** 2026-05-30
- **Context:** `docs/customization.md` advertised custom-handler registration as `HandlerRegistry::register(name, instance)` from a `ServiceProvider`, implying the registry is resolvable from the container. But `BridgeServiceProvider` built `DispatchService` with a `new HandlerRegistry` constructed *inside* the bind closure â€” never bound to the container. So `app(HandlerRegistry::class)` / `afterResolving(HandlerRegistry::class, â€¦)` could never reach the instance the dispatch loop actually used; the only working path was re-binding `DispatchService` wholesale and duplicating its constructor wiring (subscriptions / agents / registries), which is fragile across upgrades. A peer integrator reported the only documented handler-extension point as non-functional. (Filed alongside a separate, low-priority echo FR â€” per-agent self-echo for a shared upstream identity â€” tracked independently.)
- **Decision:** Bind `HandlerRegistry` as a container **singleton** in `BridgeServiceProvider`; `DispatchService` resolves it from the container (`$this->app->make(HandlerRegistry::class)`) instead of constructing its own. Operators register custom handlers via `afterResolving(HandlerRegistry::class, fn ($r) => $r->register('x', new XHandler))` in a `ServiceProvider` â€” the documented path now works against the **exact** instance the dispatcher uses, with no provider-ordering requirement (the callback is registered at boot, before the first webhook resolves the registry). `HandlerRegistry` carries no per-request state (the four shipped handlers are stateless), so a per-process singleton is correct â€” unlike the config-derived `SubscriptionRegistry` / `AgentRegistry`, which is exactly why `DispatchService` itself stays `bind` (rebuilt per request from `config('bridge.config_dir')`).
- **Alternatives considered:**
  - **Config-driven handler map** (`config/bridge.php => 'handlers' => ['x' => XHandler::class]` merged at build time). Rejected as the fix: it is a *second* registration mechanism beyond the reported bug, and a `name => FQCN` map cannot express handlers that need constructor dependencies (an API client, a configured path) â€” which the `afterResolving` path expresses naturally as `new XHandler($dep)`. The per-agent `classifier.class` extension is config-driven because it is per-agent; handlers are one global registry, so DI is the better fit. Can be added later if declarative, dependency-free registration becomes desirable.
  - **Re-bind `DispatchService` wholesale** (the prior documented workaround). Rejected: it duplicates the bridge's constructor wiring in operator code, so any future change to `DispatchService`'s dependencies silently breaks every operator override.
- **Consequences:** `HandlerRegistry` is a process singleton (built once per FPM worker, mirroring the `ClassifierResolver` per-process instance cache); `afterResolving` fires once on first resolution and the dispatcher reuses that instance. Tests that construct `DispatchService` directly with `new HandlerRegistry` are unaffected (they bypass the container). Files: `app/Providers/BridgeServiceProvider.php`, `docs/customization.md` (registration walkthrough + the extension-point table), `tests/Feature/Providers/BridgeServiceProviderTest.php` (regression test).

## DL-005 â€” Per-agent echo for a shared upstream identity via `ClassifyResult::reattributedActor`

- **Date:** 2026-05-30
- **Context:** `shared_identities` (DL-002) lets several agents run under one upstream account (e.g. four agents sharing one GitHub `sender.id` because the platform forces shared credentials). For such an account the registry resolves `Actor.name = null` on purpose, deferring attribution to a custom classifier. But echo suppression runs **before** `classify()` and matches on `(actor.name | actor.id)`, so for a shared identity the only working lever is `treat_as_echo_ids` on the raw id â€” which is **all-or-nothing**: list the id and *every* agent's view of the account is suppressed (the whole inbox dies); omit it and no self-echo happens at all (each agent sees its own writes echoed back). There is no per-agent middle ground, because the true author isn't known until the classifier re-attributes â€” and the classifier (resolved per-FQCN, shared across agents) is never told which agent it's serving. A peer integrator running the cross-cutting "subscribes-to-everything" agent reported every comment/PR it posts pinging back into its own inbox: continuous wasted attention + a latent self-react risk, defeating a guarantee the bridge advertises. DL-002 had already named this re-attribution layer as the intended completion point ("`null` name preserved byte-for-byte so that re-attribution layer is untouched").
- **Decision:** Add an optional `?Actor $reattributedActor` to `ClassifyResult` (additive, defaults null). A classifier that recovers the true author (FROM:-line / repo-scope / any operator convention) returns it there; **after** `classify`, `DispatchService` re-runs the **same** per-agent echo check (`EchoSuppression::default` over `identity.self` + `treat_as_echo` + `treat_as_echo_ids`) on the reattributed actor and marks the dispatch done (suppressed) when it's the serving agent's own write. The classifier reports *who authored the event* (content analysis); the dispatcher decides *is that me?* (per-agent policy) â€” so one shared classifier instance yields per-agent self-echo, a different shared-id agent's write still surfaces (its recovered name isn't the serving agent's self), and classifiers still **do not filter** (the documented invariant holds â€” they re-attribute, the dispatcher filters). Null `reattributedActor` is a no-op: every shipped classifier leaves it null, so behaviour is unchanged unless an operator opts in.
- **Alternatives considered:**
  - **Pass `AgentConfig` / `selfIdentity` into `classify()` (the FR's option 1).** Rejected: it is a **breaking** change to the `Classifier` contract (every custom classifier + both shipped ones must change signature, and it breaks the deliberate v0.11 Python-parity of the signature), and it moves the echo *decision* into each classifier â€” contradicting the documented "echo suppression has already happened â€” do not re-filter inside classify" invariant and inviting per-agent branching inside a per-FQCN-cached instance. It is strictly more powerful (a classifier could shape intents per agent), but that power isn't needed for this gap and costs the contract + the separation of concerns. The additive field expresses exactly the need with zero breakage.
  - **A second, separate echo-config concept evaluated post-classify.** Rejected: there is no new policy to express â€” the agent's existing `identity.self` / `treat_as_echo` already say "what is me." Reusing the same predicate on a better-attributed actor is the whole point; a parallel config would be a second source of truth for one question.
  - **Mutate `Intent.actor` and have the dispatcher re-derive the author from the intents.** Rejected: echo is an *event-level* decision (one event, one true author) and must also cover the target-only / no-intent result; reading it off per-intent actors is ambiguous and misses those cases. A single top-level field on the result is unambiguous.
  - **Defer (document as a known gap).** Rejected: it leaves a feature the bridge ships (`shared_identities`) holed for every shared-identity deployment, the fix is small and proportionate (one optional field + one dispatcher branch), and it's evidence-driven (a concrete deployment feels the pain now) rather than speculative.
- **Consequences:** `ClassifyResult` gains a third optional constructor arg; `DispatchService` gains one post-classify branch (reusing `isEcho`). The pre-classify echo gate is unchanged â€” the post-classify check is a refinement that fires only when attribution improved. The **`treat_as_signal` allowlist is intentionally NOT re-run** post-classify here (the FR scoped to echo; a shared-identity + non-empty `treat_as_signal` interaction â€” where the null-name event is dropped pre-classify before it can be re-attributed â€” is a separate, currently-unexercised concern, left for when a deployment hits it). A bridge-shipped re-attribution helper (a base classifier doing FROM:-line parsing) is a possible future DX follow-up, deliberately out of scope. Files: `app/Bridge/Dispatch/ClassifyResult.php`, `app/Bridge/Dispatch/DispatchService.php`, `docs/customization.md` (Â§ Per-agent echo for a shared upstream identity), `docs/multi-agent.md` (Path C), `tests/Feature/Dispatch/DispatchServiceTest.php` + `tests/Fixtures/ReattributingClassifier.php`.
